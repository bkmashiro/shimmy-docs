# 第二章：Linux 与内核

> 上一章我们知道了程序是什么。本章解答：谁来管理这些程序？多个程序同时运行时，它们如何不互相干扰？

## 2.1 为什么需要操作系统

想象一下没有操作系统的世界：你的程序直接控制 CPU、RAM、硬盘和网卡。

这会带来三个根本性问题：

**问题 1：硬件太复杂**  
每块硬盘的驱动方式都不同。没有操作系统，你的程序需要为每种硬盘写不同的代码。

**问题 2：无法共享资源**  
CPU 一次只能运行一件事。如果你的程序独占 CPU，其他程序就得等你结束。

**问题 3：无法安全隔离**  
你的程序和病毒程序都能访问相同的内存。病毒可以直接覆盖你的数据。

**操作系统**（Operating System，OS）解决了这三个问题：

1. **硬件抽象**：提供统一接口，程序用 `write(fd, data, len)` 写文件，不管底层是 SSD 还是 HDD。
2. **资源调度**：在多个程序之间分时共享 CPU、内存、I/O 设备。
3. **安全隔离**：每个程序只能访问自己的内存，不能干扰其他程序。

## 2.2 内核（Kernel）：操作系统的核心

**内核**是操作系统中直接与硬件交互的部分，始终运行在内存中，掌管一切资源。

Linux 内核是一个巨大的 C 程序（约 2500 万行代码），负责：
- 管理 CPU 调度（决定哪个程序在哪个时间片运行）
- 管理物理内存（把物理内存分配给各个进程）
- 管理文件系统（读写磁盘，维护目录结构）
- 管理网络（TCP/IP 协议栈）
- 管理设备驱动（与硬件通信）

### CPU 特权级（Privilege Rings）

CPU 本身设计了多个**特权级**（privilege levels），也叫**保护环**（protection rings），用来区分谁有权执行什么操作。

在 x86-64 架构上：

```
┌─────────────────────────────────────────────────────────────────┐
│                      CPU 特权级                                  │
│                                                                  │
│  Ring 0 (最高特权)                                               │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │                    内核模式（Kernel Mode）                │   │
│  │  • 可执行所有 CPU 指令                                    │   │
│  │  • 可访问所有内存地址                                     │   │
│  │  • 可直接操作硬件（I/O 端口）                             │   │
│  │  • Linux 内核运行于此                                     │   │
│  └──────────────────────────────────────────────────────────┘   │
│                          ▲                                       │
│               系统调用   │   中断/异常                           │
│                          ▼                                       │
│  Ring 3 (最低特权)                                               │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │                   用户模式（User Mode）                   │   │
│  │  • 只能执行非特权指令                                     │   │
│  │  • 只能访问自己的虚拟内存                                 │   │
│  │  • 不能直接访问硬件                                       │   │
│  │  • 你写的所有程序运行于此                                 │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  (Ring 1, Ring 2 存在但 Linux 不使用，留给虚拟化等用途)          │
└─────────────────────────────────────────────────────────────────┘
```

**为什么要有特权级？**

如果所有程序都运行在 Ring 0，任何程序都可以：
- 覆盖内核代码（系统崩溃）
- 读取其他进程的内存（泄露密码）
- 关闭中断（死锁整个系统）

Ring 3 程序尝试执行 Ring 0 指令时，CPU 会触发一个**保护异常**，内核接管并终止该程序。

当程序需要内核的服务时（比如写文件），它通过**系统调用**从 Ring 3 跳转到 Ring 0——这正是第三章的主题。

## 2.3 什么是进程（Process）

**进程**是操作系统运行程序的基本单位。每次你运行一个程序，操作系统就创建一个进程。

进程不只是程序代码——它包含程序运行时的所有状态：
- 虚拟地址空间（程序认为自己独占的内存）
- 程序计数器（RIP 的当前值）
- 寄存器状态
- 打开的文件列表
- 信号处理器
- 用户 ID（这个进程以谁的身份运行）

### 进程的虚拟地址空间

每个进程都有自己的**虚拟地址空间**（virtual address space）——一个它认为自己独占的、连续的内存空间。在 64 位 Linux 上，每个进程都以为自己有 128TB 的内存可用（尽管实际物理内存可能只有 8GB）。

```
虚拟地址空间（进程的视角，64位 Linux）

0xFFFFFFFFFFFFFFFF ┌────────────────────────────┐
                   │      内核空间 (128TB)        │ ← 只有内核能访问
0xFFFF800000000000 ├────────────────────────────┤
                   │                            │
                   │   (无效地址，访问 → 崩溃)  │
                   │                            │
0x00007FFFFFFFFFFF ├────────────────────────────┤
                   │    栈 (Stack)               │ ← 向低地址增长
                   │    [局部变量, 函数返回地址]  │
                   ├────────────────────────────┤
                   │         ↕ 增长空间          │
                   ├────────────────────────────┤
                   │    堆 (Heap)                │ ← 向高地址增长
                   │    [malloc/new 分配的内存]  │
                   ├────────────────────────────┤
                   │    .bss (未初始化全局变量)   │
                   ├────────────────────────────┤
                   │    .data (初始化全局变量)    │
                   ├────────────────────────────┤
                   │    .rodata (字符串常量等)    │
                   ├────────────────────────────┤
0x0000000000401000 │    .text (代码)             │ ← CPU 在这里执行
0x0000000000400000 ├────────────────────────────┤
                   │    ELF Header/程序头表       │
0x0000000000000000 └────────────────────────────┘
                   (NULL, 访问 → 段错误 SIGSEGV)
```

**栈**（Stack）是函数调用时使用的内存区域：每次调用函数，就在栈上压入一个**栈帧**（stack frame），包含局部变量和返回地址；函数返回时，弹出栈帧。RSP 寄存器始终指向栈顶。

**堆**（Heap）是动态内存分配的区域：调用 `malloc()`/`new` 时，从这里分配内存；调用 `free()`/`delete` 时，归还。

### 页表与虚拟内存：为什么进程 A 看不到进程 B 的内存

每个进程的虚拟地址空间是假的——实际上物理内存是共享的。那么 CPU 如何把虚拟地址翻译成物理地址？

**页表**（Page Table）是操作系统为每个进程维护的映射表，把虚拟地址分成 4KB 的**页**（page），每页映射到不同的物理地址。

```
进程 A 的页表:                  进程 B 的页表:
虚拟页 0x400 → 物理页 0x1A0     虚拟页 0x400 → 物理页 0x7F3
虚拟页 0x401 → 物理页 0x2B1     虚拟页 0x401 → 物理页 0x8C4
...                             ...

       进程 A              物理内存              进程 B
  虚拟 0x400000           ┌──────────┐       虚拟 0x400000
  (A 的代码)              │ 物理0x1A0│ ← A       (B 的代码)
       │           ┌─────►│   A代码  │           │
       │           │      └──────────┘           │
       │           │      │ 物理0x7F3│ ← B ◄─────┘
       └───页表────┘      │   B代码  │
                          └──────────┘
```

CPU 内有一个叫 **MMU**（Memory Management Unit，内存管理单元）的硬件，自动完成虚拟地址→物理地址的翻译。当切换进程时，内核切换页表，让 CPU 使用新进程的映射。

**结果：进程 A 永远只能访问其自己的物理页，完全看不到进程 B 的内存。** 任何越界访问都会触发 MMU 的**页错误**（page fault），内核接管并发送 SIGSEGV 信号给违规进程。

## 2.4 进程树与 init 进程

Linux 中所有进程都有**父进程**（parent process），形成一棵树。树的根是 **init 进程**（PID 1），它是内核启动后运行的第一个用户态进程。

```
PID 1: init (systemd)
├── PID 100: systemd-journald
├── PID 200: sshd (SSH 服务)
│   └── PID 501: bash (你的 shell)
│       └── PID 502: python3 hello.py (你运行的程序)
│           └── PID 503: ... (子进程，如果程序创建了的话)
├── PID 300: nginx (Web 服务器)
│   ├── PID 301: nginx worker
│   └── PID 302: nginx worker
└── ...
```

每个进程有一个唯一的 **PID**（Process ID，进程 ID）。

新进程通过 **`fork()`** 系统调用创建：父进程调用 fork()，内核创建父进程的几乎完整副本（新 PID，但共享相同的代码和数据）。通常子进程紧接着调用 **`execve()`** 系统调用，用新程序替换自身。这个 fork+exec 模式是 Linux 上创建进程的标准方式。

## 2.5 /proc：内核数据的虚拟文件系统

**`/proc`** 是一个特殊的文件系统，叫 **procfs**（process filesystem）。它不存储在硬盘上，而是由内核动态生成——每次你读取 `/proc` 里的文件，内核就实时生成那个文件的内容。

```
/proc/
├── 1/               ← PID 1 (init) 的信息
│   ├── cmdline      ← 启动命令行
│   ├── environ      ← 环境变量（AWS 凭证就在这里！）
│   ├── maps         ← 内存映射（地址空间布局）
│   ├── fd/          ← 打开的文件描述符
│   └── status       ← 进程状态信息
├── 1234/            ← PID 1234 的信息（类似结构）
├── self/            ← 当前进程的信息（软链接到 /proc/<自己的PID>/)
│   ├── environ      ← ⚠️ 危险！包含当前进程的所有环境变量
│   └── maps         ← ⚠️ 危险！泄露内存布局，破解 ASLR
├── cpuinfo          ← CPU 信息
├── meminfo          ← 内存使用情况
└── version          ← Linux 内核版本
```

`/proc/self/environ` 是一个严重的安全隐患：AWS Lambda 把 AWS 访问密钥以环境变量的形式传递给函数（`AWS_ACCESS_KEY_ID`、`AWS_SECRET_ACCESS_KEY`），这些敏感信息会出现在 `/proc/self/environ` 中！恶意学生代码只需读取这个文件就能窃取云账户的访问权限。

## 2.6 文件描述符（File Descriptors）

**文件描述符**（File Descriptor，fd）是内核用来追踪进程打开的资源的整数句柄。"资源"不只是普通文件，还包括管道、套接字、终端等一切可以读写的东西。

每个进程启动时，默认有三个已打开的文件描述符：

| fd | 名称 | 说明 |
|----|------|------|
| **0** | stdin（标准输入） | 通常连接到终端键盘 |
| **1** | stdout（标准输出） | 通常连接到终端屏幕 |
| **2** | stderr（标准错误） | 通常连接到终端屏幕（用于错误信息） |

当你调用 `open("/etc/passwd", O_RDONLY)` 时，内核返回一个新的文件描述符，比如 `3`。之后 `read(3, buf, 100)` 就通过这个描述符读取文件内容。

```c
int fd = open("/etc/passwd", O_RDONLY);  // fd = 3
char buf[256];
read(fd, buf, sizeof(buf));              // 通过 fd 3 读取
close(fd);                               // 关闭，释放 fd 3
```

## 2.7 Shell 与管道

**Shell**（命令行解释器，如 bash、zsh）本身就是一个普通的用户态进程。它读取你输入的命令，然后使用 fork+exec 运行它们。

**管道**（pipe）是进程间通信的基本机制：

```
ls | grep foo

               ┌─────────────────────────────────────────────────┐
               │             shell 做了什么                       │
               │                                                  │
               │  1. pipe(fds) → fds[0]=读端(fd 3), fds[1]=写端(fd 4)│
               │  2. fork → 子进程 A (ls)                         │
               │     子进程 A: 把 fd 4 复制到 stdout(fd 1)         │
               │     子进程 A: execve("/bin/ls")                  │
               │     ls 的输出写入管道写端                         │
               │  3. fork → 子进程 B (grep)                       │
               │     子进程 B: 把 fd 3 复制到 stdin(fd 0)          │
               │     子进程 B: execve("/bin/grep", "foo")         │
               │     grep 从管道读端读取 ls 的输出                 │
               └─────────────────────────────────────────────────┘

ls (写) ──[pipe]──► grep (读)
```

管道在内核中是一块缓冲区（通常 64KB），写端进程写入，读端进程读取。这是 Linux "一切皆文件"哲学的体现——管道也通过文件描述符访问。

## 2.8 Linux 权限模型

Linux 是多用户系统，通过 **uid**（User ID）和 **gid**（Group ID）区分用户，实现权限隔离。

**UID 0 是 root 用户**，拥有几乎所有权限。普通用户（uid >= 1000）权限受限。

```bash
$ id
uid=1000(alice) gid=1000(alice) groups=1000(alice),27(sudo)
```

### 文件权限

每个文件有三组权限（读 r、写 w、执行 x），分别对应：文件所有者（user）、文件所属组（group）、其他所有人（others）：

```
-rwxr-xr--  1 alice  alice  1234  Mar 1 10:00  hello
 ↑↑↑↑↑↑↑↑↑
 │└┬┘└┬┘└┬┘
 │ │  │  └── others: r-- (只读)
 │ │  └───── group:  r-x (读+执行)
 │ └──────── owner:  rwx (读+写+执行)
 └────────── 文件类型: - = 普通文件
```

### Linux Capabilities

传统 Unix 是全有或全无：要么是 root（全能），要么不是 root（受限）。这太粗糙了。

**Linux Capabilities** 把 root 的特权分成 40+ 个细粒度能力，可以单独授予或撤销：

| Capability | 允许的操作 |
|-----------|---------|
| CAP_NET_ADMIN | 配置网络接口 |
| CAP_SYS_ADMIN | 各种系统管理操作（非常强大） |
| CAP_SYS_CHROOT | 调用 chroot() |
| CAP_NET_RAW | 创建原始套接字 |
| CAP_SYS_PTRACE | 使用 ptrace() |

在 AWS Lambda 中，`sbx_user1051` 用户（uid 993）没有任何 capabilities，这严重限制了可以使用的沙箱技术，也是第六章要深入分析的核心问题。

## 2.9 程序为什么不能直接访问硬件

把本章的所有知识综合起来，现在可以回答这个根本问题：

**程序不能直接访问硬件，因为 CPU 硬件层面的特权级机制强制执行了这一点。**

具体来说：

1. **程序运行在 Ring 3**（用户模式），CPU 硬件限制了这个模式下可执行的指令集。
2. **I/O 指令**（如 `IN`、`OUT`，直接访问硬件端口）只能在 Ring 0 执行。在 Ring 3 执行 → CPU 触发保护异常 → 内核 SIGKILL 该进程。
3. **内存访问**受 MMU 控制。程序只能访问其页表中映射的内存，越界 → 页错误 → SIGSEGV。
4. **需要硬件服务时**，程序必须通过系统调用请求内核代劳。内核在内部完成硬件操作后，把结果返回给程序。

```
程序想写文件:

Ring 3: printf("hello") → write() → syscall指令
              ↓（CPU 切换到 Ring 0）
Ring 0: 内核验证权限
        内核操作文件系统驱动
        内核通过 I/O 指令与存储设备通信
              ↓（CPU 切换回 Ring 3）
Ring 3: write() 返回，程序继续运行
```

**这就是操作系统存在的根本原因**：在裸 CPU 和程序之间充当可信中间人，安全、公平地共享硬件资源。

## 小结

本章建立了以下概念链：

```
CPU Ring 0 (内核) vs Ring 3 (用户)
    ↓
进程 = 独立虚拟地址空间 + 状态
    ↓
页表 → 虚拟内存 → 进程间隔离
    ↓
/proc → 内核数据虚拟化（含危险的 environ）
    ↓
文件描述符 → 一切 I/O 的抽象
    ↓
uid/gid/capabilities → 权限模型
```

**核心要点**：
1. 内核运行在特权 Ring 0，用户程序运行在受限 Ring 3
2. 每个进程有独立的虚拟地址空间，页表保证隔离
3. `/proc/self/environ` 暴露环境变量（包括 AWS 凭证）
4. Lambda 用户没有 capabilities，这使得标准沙箱工具大多失效
