## 摘要

本文系统记录了最先进的系统调用拦截机制 zpoline 和 lazypoline 中五类基础设计和实现缺陷，统称为"系统调用插桩陷阱"（P1–P5）。基于这些发现，作者设计了 K23——一个即插即用的 x86-64 系统调用拦截器，采用离线阶段（使用 SUD 记录系统调用位置）结合在线阶段（选择性重写 + ptrace 处理启动 + SUD 兜底）的方式，解决所有已识别陷阱，同时维持与 zpoline 相当的性能。K23-default 在 Web 服务器基准测试中达到原生吞吐量的 98.62%——优于 lazypoline（98.26%），仅略低于 zpoline（98.93%）。

## 核心思想

> **💡 什么是"攻击面"（attack surface）？**
> 攻击面是指系统中所有可能被攻击者利用的入口点的总和。就像一座城堡，门越多、窗户越多，攻击面就越大。减小攻击面意味着减少这些潜在入口——例如，禁止不必要的系统调用，就像封死城堡里不必要的侧门。

- **五类陷阱（P1–P5）**：每类都有概念验证漏洞利用（PoC）。任何使用 LD_PRELOAD 的机制都容易受到 P1a 攻击（在 execve 中清除环境变量），这一发现延伸到所有基于 LD_PRELOAD 的工具。
- **离线阶段**：使用良性输入运行目标程序，通过基于 SUD 的 `libLogger` 记录有效的 `(内存区域, 偏移量)` 对，对应 `syscall`/`sysenter` 指令位置。真实应用中仅观察到 7–92 条唯一指令。
- **在线阶段**：(1) 从第一条指令开始用 ptrace 覆盖启动过程和 vDSO；(2) libK23 通过伪系统调用握手接管；(3) 一次性选择性重写仅经过预验证的位置；(4) SUD 兜底处理离线阶段遗漏的位置。
- **哈希集合 vs. 位图**：K23 将 zpoline 的全地址空间位图（MB 级别）替换为仅包含离线记录地址的 robin-hood 哈希集合（内存占用可忽略不计），解决 P4b。
- **K23 是唯一解决所有九个陷阱的机制**：zpoline 和 lazypoline 各自至少漏掉三个。

## 与 Shimmy 的关联

K23 是目前安全性最完整的系统调用插桩机制，其发现对 Shimmy 的设计有直接影响。最关键的一点：**LD_PRELOAD 不可信**——任何使用 LD_PRELOAD 的沙箱机制（包括 zpoline、lazypoline、DynamoRIO 的注入模式）都可以通过以干净环境调用 `execve` 来绕过。对于 Shimmy 的自托管层（SUD 和 ptrace 可用），K23 的离线+在线模型是推荐方案。对于 Lambda（SUD 需要内核 5.11+ 而 ptrace 被阻塞），这些技术不能直接使用，但 K23 的陷阱分析揭示了 Shimmy Lambda 模式沙箱无法声明拥有的安全属性。

陷阱分析也验证了 Shimmy 对 Lambda 上基于 WASM 沙箱的偏好：WASM 提供的隔离不会被 P1–P5 中的任何陷阱绕过，因为 WASM 指令集中根本没有 syscall 指令。

## 详细说明

### 五类陷阱

**P1 — 插桩绕过**：
- **P1a**（zpoline + lazypoline）：通过清除环境变量或以空环境调用 `execve` 可绕过基于 LD_PRELOAD 的注入。拦截器库永远不会被子进程加载。
- **P1b**（仅 lazypoline）：攻击者可以调用 `prctl(PR_SET_SYSCALL_USER_DISPATCH, SECCOMP_USER_DISPATCH_OFF)` 直接禁用 SUD，消除慢速路径兜底。

**P2 — 系统调用遗漏**：
- **P2a**：zpoline 的静态反汇编无法在变长 x86-64 代码中找到所有 `syscall`/`sysenter` 位置。lazypoline 通过动态发现改进了这一点，但仍有盲点。
- **P2b**（zpoline + lazypoline）：两者都无法拦截库加载前或加载期间发起的系统调用，以及基于 vDSO 的系统调用。即使是 `ls` 命令，在拦截器库加载前也会发起 100+ 次系统调用。

> **💡 什么是 vDSO？**
> vDSO（虚拟动态共享对象）是内核注入到每个进程地址空间的一小段代码。某些高频系统调用（如 `gettimeofday`）通过 vDSO 直接在用户态执行，完全不经过 `syscall` 指令，因此无法被 zpoline 式的二进制重写拦截到。

**P3 — 指令误识别**：
- **P3a**：zpoline 的静态反汇编可能将看起来像 `syscall`/`sysenter` 操作码的字节实际上是数据或其他指令的一部分误改写，导致代码或数据损坏。
- **P3b**：lazypoline 避免了静态反汇编 bug，但容易受到控制流劫持攻击：攻击者将执行流重定向到包含 `0x0f 0x05` 字节的数据页，导致 SUD 将这些字节改写为"系统调用位置"。

**P4 — NULL 访问终止陷阱**：
- **P4a**（lazypoline）：在虚拟地址 0 处映射了蹦床，但不检查意外调用者。本来会导致段错误的 NULL 代码指针 bug 现在悄无声息地跳入蹦床——调试的噩梦。
- **P4b**（zpoline）：使用全虚拟地址空间位图来验证调用者，在多进程场景下造成显著内存开销。

**P5 — 运行时重写陷阱**（lazypoline）：
- 非原子的两字节替换（可被另一个核心上的并发执行撕裂）
- 重写后未刷新指令缓存
- 无指令序列化屏障
- 不可靠的页面权限保存/恢复（未考虑 XOM 页面）

**陷阱覆盖矩阵**：

| 陷阱 | zpoline | lazypoline | K23 |
|---------|---------|------------|-----|
| P1a LD_PRELOAD 绕过 | ✗ | ✗ | ✓ |
| P1b SUD 通过 prctl 禁用 | ✓ (不适用) | ✗ | ✓ |
| P2a 反汇编遗漏 | ✗ | ✓ | ✓ |
| P2b 加载前/vDSO 遗漏 | ✗ | ✗ | ✓ |
| P3a 反汇编误识别 | ✗ | ✓ (不适用) | ✓ |
| P3b 攻击诱导误识别 | ✓ (不适用) | ✗ | ✓ |
| P4a NULL 执行检查 | ✓ | ✗ | ✓ |
| P4b 位图内存开销 | ✗ | ✓ (不适用) | ✓ |
| P5 运行时重写问题 | ✓ | ✗ | ✓ |

### K23 架构

**离线阶段**：
1. 通过 LD_PRELOAD 注入 `libLogger`（基于 SUD）
2. 使用良性输入运行程序；SIGSYS 信号处理程序为每条触发的指令提取 `(内存区域路径, 偏移量)`
3. 仅记录来自预期可执行非可写区域的指令（而非 JIT 生成的代码）
4. 将日志目录设为不可变，防止篡改

观察到的唯一指令数：`pwd`/`ls`/`cat` 等——7–13 条；nginx/lighttpd/redis/sqlite——20–92 条。

**在线阶段**：

1. **ptrace 追踪器**（跨进程，从第一条指令开始）：
   - 从最开始拦截所有系统调用
   - 禁用 vDSO（强制 vDSO 调用使用 `syscall`/`sysenter`）→ 解决 P2b
   - 拦截 `execve` 并强制向子进程注入 LD_PRELOAD → 解决 P1a

2. **libK23 接管**（通过伪系统调用握手）：
   - libK23 发起一个伪系统调用（不可能的系统调用号）；内核将其路由到 ptrace 追踪器
   - ptrace 追踪器通过 `process_vm_writev` 传递累积状态，然后解除附加
   - libK23 发起第二个伪系统调用确认解除附加

3. **一次性选择性重写**：
   - 在虚拟地址 0 处安装蹦床（带 XOM PKU 保护）
   - 仅重写离线日志中预验证的位置 → 解决 P3a、P3b、P5
   - 正确保存/恢复页面权限

4. **SUD 兜底**：
   - 仅作为兜底机制启用 SUD，处理离线阶段遗漏的位置 → 解决 P2a
   - libK23 拦截任何试图禁用 SUD 的 `prctl` 调用 → 解决 P1b

5. **蹦床入口检查**：
   - 使用离线记录地址的 robin-hood 哈希集合（而非全地址空间位图）→ 解决 P4a、P4b
   - 未知调用者触发进程中止

### 评估

**硬件**：DELL Precision 7960，12 核 Intel Xeon w5-3425 3.20 GHz，64 GB RAM，Ubuntu 22.04，Linux 6.8.0-85。

**微基准测试**（1 亿次不存在的系统调用）：

| 机制 | 相对原生开销 |
|-----------|---------------------|
| zpoline-default | 1.1267× |
| SUD（仅初始化）| 1.2269× |
| K23-default | 1.2788× |
| lazypoline | 1.3801× |
| SUD（完整拦截）| 15.3022× |

K23-default 比 lazypoline 更快，因为它利用了内核在系统调用执行时会破坏 `rcx` 和 `r11` 的特性，节省了两个寄存器的保存操作。

**宏观基准测试**（nginx、lighttpd、redis、sqlite——几何平均）：

| 机制 | 相对原生吞吐量 |
|-----------|-----------------------|
| zpoline-default | 98.93% |
| K23-default | 98.62% |
| lazypoline | 98.26% |
| SUD | 56.70% |

K23-default 在解决所有陷阱的同时，达到了与 zpoline 几乎相同的吞吐量。

### 局限

1. **仅限 x86-64 Linux**：依赖 SUD（Linux 特有）和 zpoline 风格的变长指令重写
2. **离线阶段覆盖率**：需要具有代表性的工作负载；并非所有应用都有结构化基准测试
3. **SUD 固有代价**：即使不进行拦截，启用 SUD 也会带来约 1.23× 开销，这是 K23 无法消除的内核层面代价
4. **未实现内部状态保护**：论文假设开发者会部署 PKU 或类似机制；原型未包含此功能

### 术语表

- **K23**：结合离线阶段（基于 SUD 的日志记录）+ 在线阶段（选择性重写 + ptrace 启动 + SUD 兜底）的系统调用拦截器
- **离线阶段**：生产前分析，使用 SUD 记录有效系统调用指令位置
- **在线阶段**：使用 ptrace（启动）、选择性二进制重写（热路径）和 SUD（兜底）的运行时拦截
- **P1–P5**：本文识别的五类系统调用插桩陷阱
- **libLogger**：K23 的离线阶段插桩库；使用 SUD 记录 `(区域, 偏移量)` 对
- **ptracer**：K23 的跨进程组件；从第一条指令开始活跃，处理启动时系统调用和 vDSO
- **SUD（Syscall User Dispatch）**：Linux 5.11+ 内核接口，在系统调用时触发 SIGSYS
- **robin-hood 哈希集合**：K23 在蹦床入口检查调用者有效性时使用的内存高效哈希集合
- **XOM（仅执行内存）**：蹦床的内存保护模式；保留 NULL 读写故障
