# Clair Obscur: The Light and Shadow of System Call Interposition — From Pitfalls to Solutions with K23

**Authors:** Jesús María Gómez Moreno, Vissarion Moutafis, Antreas Dionysiou, Fernando Kuipers, Georgios Smaragdakis, Bart Coppens, Alexios Voulimeneas (TU Delft / Ghent University)  **Venue:** Middleware  **Year:** 2025

## Summary

This paper systematically documents five categories of fundamental design and implementation flaws in the state-of-the-art syscall interception mechanisms zpoline and lazypoline, collectively termed "System Call Interposition Pitfalls" (P1–P5). Based on these findings, the authors design K23, a plug-and-play x86-64 syscall interceptor that uses an offline phase (logging syscall sites with SUD) combined with an online phase (selective rewriting + ptrace for startup + SUD fallback) to address all identified pitfalls while maintaining performance comparable to zpoline. K23-default achieves 98.62% of native throughput on web server benchmarks — better than lazypoline (98.26%) and only slightly behind zpoline (98.93%).

## Key Ideas

- **Five pitfall categories (P1–P5)**: Each backed by a PoC exploit. Any mechanism using LD_PRELOAD is vulnerable to P1a (clearing env vars in execve), which extends the finding to all LD_PRELOAD-based tools.
- **Offline phase**: Run the target program with benign inputs using a SUD-based `libLogger` to record valid `(memory_region, offset)` pairs for `syscall`/`sysenter` instructions. Only 7–92 unique instructions observed for real applications.
- **Online phase**: (1) ptrace from first instruction to cover startup and vDSO; (2) libK23 takes over via fake syscall handoff; (3) one-time selective rewriting of only pre-validated sites; (4) SUD fallback for sites missed by offline phase.
- **Hash set vs. bitmap**: K23 replaces zpoline's full-address-space bitmap (MB-scale) with a robin-hood hash set containing only offline-logged addresses (negligible memory), solving P4b.
- **K23 is the only mechanism addressing all nine pitfalls**: zpoline and lazypoline each miss at least three.

## Relevance to Shimmy

K23 is the most security-complete syscall interposition mechanism available, and its findings have direct implications for Shimmy's design. The most critical: **LD_PRELOAD cannot be trusted** — any sandboxing mechanism using LD_PRELOAD (including zpoline, lazypoline, DynamoRIO's injection mode) can be bypassed by calling `execve` with a clean environment. For Shimmy's self-hosted tier (where SUD and ptrace are available), K23's offline+online model is the recommended approach. For Lambda (where SUD requires kernel 5.11+ and ptrace is blocked), these techniques are not directly available, but K23's pitfall analysis informs what security properties Shimmy's Lambda-mode sandboxing cannot claim.

The pitfall analysis also validates Shimmy's preference for WASM-based sandboxing on Lambda: WASM provides isolation that cannot be bypassed by any of the P1–P5 pitfalls because there are no syscall instructions in WASM's instruction set.

## Detailed Notes

### The Five Pitfall Categories

**P1 — Interposition Bypass**:
- **P1a** (zpoline + lazypoline): LD_PRELOAD-based injection can be bypassed by clearing environment variables or calling `execve` with an empty environment. The interceptor library is never loaded for child processes.
- **P1b** (lazypoline only): An attacker can call `prctl(PR_SET_SYSCALL_USER_DISPATCH, SECCOMP_USER_DISPATCH_OFF)` to directly disable SUD, eliminating the slow-path fallback.

**P2 — System Call Overlook**:
- **P2a**: zpoline's static disassembly fails to find all `syscall`/`sysenter` sites in variable-length x86-64 code. lazypoline improves this with dynamic discovery but still has blind spots.
- **P2b** (zpoline + lazypoline): Neither can intercept syscalls made before or during library loading, or vDSO-based syscalls. Even `ls` makes 100+ syscalls before the interceptor library is loaded.

**P3 — Instruction Misidentification**:
- **P3a**: zpoline's static disassembly may rewrite bytes that look like `syscall`/`sysenter` opcodes but are actually data or parts of other instructions, corrupting code or data.
- **P3b**: lazypoline avoids static disassembly bugs but is vulnerable to control-flow hijacking: an attacker redirects execution to a data page containing `0x0f 0x05` bytes, causing SUD to rewrite those bytes as a "syscall site."

**P4 — NULL Access Termination Pitfalls**:
- **P4a** (lazypoline): Maps a trampoline at virtual address 0 but doesn't check for unexpected callers. A NULL code pointer bug that would have segfaulted now silently jumps into the trampoline — a debugging nightmare.
- **P4b** (zpoline): Uses a full-virtual-address-space bitmap to validate callers, imposing significant memory overhead in multi-process scenarios.

**P5 — Runtime Rewriting Pitfalls** (lazypoline):
- Non-atomic 2-byte replacement (can be torn by concurrent execution on another core)
- No instruction cache flush after rewriting
- No instruction serialization barrier
- Unreliable page permission save/restore (doesn't account for XOM pages)

|---------|---------|------------|-----|
| P1a LD_PRELOAD bypass | ✗ | ✗ | ✓ |

### K23 Architecture

**Offline Phase**:
1. Inject `libLogger` (SUD-based) via LD_PRELOAD
2. Run program with benign inputs; SIGSYS signal handler extracts `(memory_region_path, offset)` for each triggered instruction
3. Only log instructions from expected executable non-writable regions (not JIT-generated code)
4. Mark log directory immutable to prevent tampering

Observed unique instruction counts: `pwd`/`ls`/`cat` etc. — 7–13 instructions; nginx/lighttpd/redis/sqlite — 20–92 instructions.

**Online Phase**:

1. **ptracer** (cross-process, first instruction):
   - Intercepts all syscalls from the very first instruction
   - Disables vDSO (forces vDSO calls to use `syscall`/`sysenter`) → solves P2b
   - Intercepts `execve` and forces LD_PRELOAD injection into child processes → solves P1a

2. **libK23 takeover** (via fake syscall handshake):
   - libK23 issues a fake syscall (impossible syscall number); kernel routes it to ptracer
   - ptracer transfers accumulated state via `process_vm_writev`, then detaches
   - libK23 issues a second fake syscall to confirm detachment

3. **One-time selective rewriting**:
   - Installs trampoline at virtual address 0 (with XOM PKU protection)
   - Rewrites only the pre-validated sites from the offline log → solves P3a, P3b, P5
   - Saves/restores page permissions correctly

4. **SUD fallback**:
   - Enables SUD as a fallback-only mechanism for sites missed by offline phase → solves P2a
   - libK23 intercepts any `prctl` call attempting to disable SUD → solves P1b

5. **Trampoline entry check**:
   - Uses a robin-hood hash set of offline-logged addresses (not a full-address-space bitmap) → solves P4a, P4b
   - Unknown callers trigger process abort

### Evaluation

**Hardware**: DELL Precision 7960, 12-core Intel Xeon w5-3425 3.20 GHz, 64 GB RAM, Ubuntu 22.04, Linux 6.8.0-85.

**Microbenchmarks** (100M non-existent syscalls):

| Mechanism | Overhead vs. native |
|-----------|---------------------|
| zpoline-default | 1.1267× |
| SUD (init only) | 1.2269× |
| K23-default | 1.2788× |
| lazypoline | 1.3801× |
| SUD (full interception) | 15.3022× |

K23-default is faster than lazypoline because it exploits that the kernel destroys `rcx` and `r11` on syscall execution, saving two register saves.

**Macrobenchmarks** (nginx, lighttpd, redis, sqlite — geometric mean):

| Mechanism | Throughput vs. native |
|-----------|-----------------------|
| zpoline-default | 98.93% |
| K23-default | 98.62% |
| lazypoline | 98.26% |
| SUD | 56.70% |

K23-default achieves nearly the same throughput as zpoline while being the only mechanism that addresses all pitfalls.

### Limitations

1. **x86-64 Linux only**: Depends on SUD (Linux-specific) and zpoline-style variable-length instruction rewriting
2. **Offline phase coverage**: Requires representative workloads; not all applications have structured benchmarks
3. **SUD inherent cost**: ~1.23× overhead from enabling SUD (even without interception) is a kernel-level cost K23 cannot eliminate
4. **No internal state protection implemented**: Paper assumes developers will deploy PKU or similar; prototype doesn't include it

### Glossary

- **K23**: Syscall interceptor combining offline phase (SUD-based logging) + online phase (selective rewriting + ptrace startup + SUD fallback)
- **Offline phase**: Pre-production analysis that records valid syscall instruction locations using SUD
- **Online phase**: Runtime interception using ptrace (startup), selective binary rewriting (hot path), and SUD (fallback)
- **P1–P5**: The five categories of syscall interposition pitfalls identified in this paper
- **libLogger**: K23's offline-phase instrumentation library; uses SUD to log `(region, offset)` pairs
- **ptracer**: K23's cross-process component; active from first instruction, handles startup syscalls and vDSO
- **SUD (Syscall User Dispatch)**: Linux 5.11+ kernel interface triggering SIGSYS on syscall invocation
- **Robin-hood hash set**: Memory-efficient hash set used by K23 to check caller validity at trampoline entry
- **XOM (eXecute-Only Memory)**: Memory protection mode for the trampoline; preserves NULL read/write faults
