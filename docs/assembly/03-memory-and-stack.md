# 第三章：内存布局与栈

> 程序运行时，内存是如何组织的？函数调用时，数据存在哪里？本章介绍进程的内存布局，重点讲解"栈"这个核心数据结构，以及函数调用背后发生的一切。

## 3.1 进程内存布局

每个运行中的程序都有自己的**虚拟地址空间**。操作系统给每个进程一种错觉：好像整个内存就是你一个人的。

在 Linux x86-64 下，一个典型进程的内存布局如下（地址从高到低）：

```
高地址（0xFFFFFFFFFFFFFFFF）
┌─────────────────────────────────┐
│         内核空间（Kernel Space） │ ← 用户程序无法访问
├─────────────────────────────────┤ ← 通常在 0x7FFFFFFFFFFF
│                                 │
│    命令行参数 & 环境变量          │
│                                 │
├─────────────────────────────────┤
│                                 │
│     栈（Stack）           ↓ 向下增长 │
│                                 │
│  ·  ·  ·  ·（空闲空间）·  ·  ·  │
│                                 │
│     堆（Heap）            ↑ 向上增长 │
│                                 │
├─────────────────────────────────┤
│         BSS 段                  │ ← 未初始化的全局/静态变量
├─────────────────────────────────┤
│         数据段（Data）           │ ← 已初始化的全局/静态变量
├─────────────────────────────────┤
│         代码段（Text）           │ ← 程序指令（只读）
├─────────────────────────────────┤ ← 通常从 0x400000 开始
│         （保留/空洞）            │
低地址（0x0000000000000000）
```

### 各个区域详解

#### 代码段（Text Segment / Code Segment）
存储程序的**机器码指令**。这个区域通常是**只读**的——操作系统不允许程序修改自己的代码（防止某类安全攻击）。

```c
int add(int a, int b) {  // 这个函数的机器码存储在代码段
    return a + b;
}
```

#### 数据段（Data Segment）
存储**已初始化的全局变量和静态变量**。

```c
int global_count = 100;      // 存储在数据段
static char prefix[] = "ID"; // 存储在数据段
```

#### BSS 段（Block Started by Symbol）
存储**未初始化的全局变量和静态变量**。BSS 段在文件中不占空间（只记录大小），程序加载时由操作系统填零。

```c
int uninitialized_array[1000]; // 存储在 BSS 段，不占可执行文件的空间
```

#### 堆（Heap）
**动态分配**的内存区域，从低地址向高地址增长。

```c
int *arr = malloc(100 * sizeof(int)); // 在堆上分配 400 字节
free(arr);                            // 释放堆内存
```

Python 里你用的所有对象（列表、字典、字符串……）都在堆上，由 Python 的垃圾回收器管理。

#### 栈（Stack）
**函数调用的核心区域**，从高地址向低地址增长（向下）。这是本章的重点。

---

## 3.2 栈：一个盘子叠盘子的数据结构

**栈**（Stack）是一种后进先出（LIFO, Last In First Out）的数据结构，就像叠放的盘子：

```
     操作        栈的状态
初始状态:         [ 空 ]
push A:          [ A ]
push B:          [ A | B ]     ← B 在顶部
push C:          [ A | B | C ] ← C 在顶部
pop:    → C     [ A | B ]     ← C 被取出
pop:    → B     [ A ]
pop:    → A     [ 空 ]
```

规则：只能从顶部（栈顶）放入（push）或取出（pop）数据。

### 在内存中，栈是倒置的

栈在内存中**向低地址增长**（向下），这让初学者常常感到困惑：

```
内存地址（高 → 低）

0x7FFF1008  [ 旧数据（已 pop 的） ]
0x7FFF1000  [ 第一个 push 的数据 ]  ← 栈底（高地址）
0x7FFF0FF8  [ 第二个 push 的数据 ]
0x7FFF0FF0  [ 第三个 push 的数据 ]  ← 栈顶（低地址）  ← rsp 指向这里
```

**`rsp`（Stack Pointer，栈指针）**始终指向当前栈顶，即最后一个压入数据的地址。

**`push` 操作**：
1. `rsp` 减 8（因为每个值是 8 字节）
2. 把数据写入 `[rsp]`（rsp 指向的内存地址）

**`pop` 操作**：
1. 从 `[rsp]` 读取数据到目标寄存器
2. `rsp` 加 8

用汇编表示：
```asm
push rax    ; 等价于：sub rsp, 8 ; mov [rsp], rax
pop  rbx    ; 等价于：mov rbx, [rsp] ; add rsp, 8
```

## 3.3 栈帧（Stack Frame）

每次调用一个函数，栈上就会分配一块新的空间，称为**栈帧**（Stack Frame）或**激活记录**（Activation Record）。这个栈帧存放该函数的：

- 局部变量
- 保存的寄存器（调用前的值）
- 返回地址（函数执行完应该跳回哪里）

`rbp`（Base Pointer，基址指针）指向**当前栈帧的底部**，作为固定的参考点（因为栈帧大小固定后，局部变量相对 `rbp` 的偏移也是固定的）。

### 栈帧结构图

假设有这样的 C 代码：

```c
int foo(int x, int y) {
    int local_a = x + y;
    int local_b = x * 2;
    return local_a + local_b;
}

int main() {
    int result = foo(3, 4);
    return 0;
}
```

当 `main` 调用 `foo(3, 4)` 时，栈的状态：

```
高地址
┌───────────────────────────────────────────┐
│              main 的栈帧                   │
│  ┌─────────────────────────────────────┐  │
│  │  main 的局部变量 (result, ...)      │  │
│  │  main 保存的寄存器                   │  │
│  └─────────────────────────────────────┘  │
│                                           │
├───────────────────────────────────────────┤  ← main 调用 foo 前
│           返回地址（call 压入）            │  ← foo 执行完要跳回 main 的哪一行
├───────────────────────────────────────────┤
│           旧的 rbp（push rbp 压入）        │  ← rbp 指向这里
├───────────────────────────────────────────┤  ← foo 的栈帧开始
│  [rbp - 4]  local_a = 7                  │
│  [rbp - 8]  local_b = 6                  │
│                                           │
│     foo 的栈帧                            │
│                                           │
└───────────────────────────────────────────┘  ← rsp 指向这里（栈顶，低地址）
低地址
```

## 3.4 函数调用时栈上发生了什么

让我们一步步追踪 `main` 调用 `foo(3, 4)` 的完整过程：

### 第一步：`main` 准备参数（放入寄存器）

```asm
; main 的代码
mov edi, 3    ; 第一个参数 x = 3，放入 edi
mov esi, 4    ; 第二个参数 y = 4，放入 esi
call foo      ; 调用 foo
```

`call` 指令做了两件事：
1. 把**返回地址**（`call` 指令之后那条指令的地址）压入栈（`push rip`）
2. 跳转到 `foo` 的地址

### 第二步：`foo` 的函数序言（Prologue）

```asm
foo:
    push rbp          ; 保存 main 的 rbp（这样 foo 返回时 main 的 rbp 能恢复）
    mov  rbp, rsp     ; 建立新栈帧：rbp = 当前 rsp
    sub  rsp, 16      ; 为局部变量分配空间（local_a 和 local_b 各 4 字节，对齐到 16）
```

此时栈的状态：

```
地址            内容
─────────────────────────────────────
rsp+16   →   [ 返回地址（call 压入）]
rsp+8    →   [ 旧 rbp（main 的 rbp）]   ← rbp 指向这里
rsp+4    →   [ local_a（待赋值）      ]
rsp      →   [ local_b（待赋值）      ]   ← rsp 指向这里
```

### 第三步：`foo` 的函数体执行

```asm
    ; int local_a = x + y;
    mov eax, edi          ; eax = x (参数 x 在 edi 中)
    add eax, esi          ; eax = x + y
    mov DWORD PTR [rbp-4], eax  ; local_a = eax（存入 rbp-4 的位置）
    
    ; int local_b = x * 2;
    mov eax, edi          ; eax = x
    imul eax, eax, 2      ; eax = x * 2
    mov DWORD PTR [rbp-8], eax  ; local_b = eax
    
    ; return local_a + local_b;
    mov eax, DWORD PTR [rbp-4]  ; eax = local_a
    add eax, DWORD PTR [rbp-8]  ; eax = local_a + local_b
    ; 结果在 eax 中，准备返回
```

### 第四步：`foo` 的函数尾声（Epilogue）和返回

```asm
    ; 函数尾声
    mov rsp, rbp   ; 恢复 rsp（丢弃局部变量占用的栈空间）
    pop rbp        ; 恢复 main 的 rbp
    ret            ; 从栈上弹出返回地址，跳回 main
```

`ret` 等价于 `pop rip`：从栈上弹出返回地址，放入 `rip`，CPU 就跳回 `main` 继续执行。

### 更紧凑的写法

函数尾声的 `mov rsp, rbp; pop rbp` 经常被简化为一条指令：

```asm
leave   ; 等价于：mov rsp, rbp ; pop rbp
ret
```

## 3.5 为什么栈向下增长

这是一个历史设计决策，有几个好处：

1. **代码段从低地址开始，堆从中间开始向上增长，栈从高地址向下增长**：这样堆和栈可以"背靠背"增长，充分利用中间的空闲空间
2. **`rsp` 总是指向最新的数据，而 `rbp` 指向栈帧底部**：`rbp` 大于 `rsp` 的设计，让局部变量地址的计算（`rbp - n`）更直观

### 一个常见的困惑

"向下增长"意味着：**push 让 rsp 变小**，**pop 让 rsp 变大**。

这和直觉相反（大多数人认为"堆上去"应该地址变大），但记住：**"顶"在汇编语言里是低地址**。

```
"栈顶"（Top of Stack）= 低地址 = rsp 指向的地方
"栈底"（Bottom of Stack）= 高地址 = 栈帧开始的地方
```

## 3.6 栈对齐（Stack Alignment）

在 x86-64 Linux 下，**`call` 指令执行前，`rsp` 必须是 16 字节对齐的**（即 `rsp % 16 == 0`）。这是 System V ABI 的要求，某些 SSE/AVX 指令要求数据 16 字节对齐。

调用约定规定：进入一个函数时（刚执行完 `call`），`rsp` 是 16 字节对齐减去 8（因为 `call` 压入了 8 字节的返回地址）。

```asm
foo:
    push rbp        ; 再压入 8 字节，rsp 现在是 16 字节对齐
    mov  rbp, rsp   ; 此时 rsp % 16 == 0
    sub  rsp, 16    ; 分配空间，保持 16 字节对齐
```

如果你分配的空间大小不是 16 的倍数，可能需要额外填充，确保 `call` 子函数前 `rsp % 16 == 0`。这个细节在手写汇编时很重要。

## 3.7 栈溢出和缓冲区溢出

了解了栈的结构，我们来理解两个重要的安全概念：

### 栈溢出（Stack Overflow）

如果递归太深（无限递归），每次函数调用都分配一个新的栈帧，栈会不断向下增长，直到超出操作系统给这个进程分配的栈空间限制，程序崩溃。

```python
# Python 中无限递归
def infinite():
    return infinite()

infinite()  # RecursionError: maximum recursion depth exceeded
```

这就是网站 stackoverflow.com 名字的由来。

### 缓冲区溢出（Buffer Overflow）

考虑这段 C 代码：

```c
void vulnerable(char *input) {
    char buffer[64];           // 64字节的局部变量，在栈上
    strcpy(buffer, input);     // 把 input 复制到 buffer（不检查长度！）
}
```

`buffer` 在栈上：`[rbp - 64]` 到 `[rbp - 1]`。

如果 `input` 的长度超过 64 字节，`strcpy` 会继续往高地址写，覆盖：
- `[rbp]`：旧的 `rbp` 值（被破坏）
- `[rbp + 8]`：**返回地址**（被攻击者控制！）

```
栈布局：
┌─────────────────────────────┐
│  buffer[0..63]              │ ← 正常数据
│  （你的64字节缓冲区）        │
├─────────────────────────────┤
│  旧 rbp                     │ ← 溢出后被覆盖
├─────────────────────────────┤
│  返回地址                   │ ← 溢出后被攻击者控制！
├─────────────────────────────┤
│  main 的栈帧 ...            │
└─────────────────────────────┘
```

攻击者可以把返回地址覆盖成任意代码的地址，劫持程序执行流。这就是最经典的栈溢出攻击，也是理解为什么现代系统需要**栈金丝雀**（stack canary）、**NX位**（不可执行栈）、**ASLR**（地址空间布局随机化）的原因。

## 3.8 用 `rsp` 和 `rbp` 追踪函数调用链

调试程序时，可以通过 `rbp` 链表重建整个调用栈：

```
当前栈帧的 rbp  →  [旧 rbp]  →  [更旧的 rbp]  →  ...
                   [返回地址]    [返回地址]
```

这就是 `gdb` 的 `backtrace`（`bt`）命令能打印调用栈的原因——它沿着 `rbp` 链向上追溯。

```
(gdb) bt
#0  foo (x=3, y=4) at add.c:2
#1  0x000000000040115a in main () at add.c:7
```

## 小结

- **进程内存**分为代码段、数据段、BSS段、堆、栈，各有用途
- **栈**是后进先出的数据结构，在内存中**向下增长**（向低地址）
- **`rsp`**（栈指针）始终指向栈顶（当前最低的有效地址）
- **`rbp`**（基址指针）指向当前栈帧的底部，作为访问局部变量的参考点
- **函数调用过程**：准备参数 → `call`（压入返回地址，跳转）→ 函数序言（保存旧rbp，建立新栈帧）→ 函数体 → 函数尾声（恢复栈帧）→ `ret`（弹出返回地址，跳回）
- **缓冲区溢出**：理解栈布局是理解这类经典漏洞的基础
- **栈对齐**：x86-64 要求 `call` 前 `rsp` 是 16 字节对齐的
