# 内核态与用户态

> 你的程序和操作系统内核都在同一台计算机上运行，但它们的"权限级别"天壤之别。本文深入解释这个差异从哪里来、为什么存在，以及这是所有沙箱技术不得不面对的根本约束。

## 前置知识

- 知道什么是 CPU 和内存
- 了解"操作系统"和"程序"的基本区别
- 最好已读过《进程与内存空间》

## 你将学到

- CPU 特权级（Ring 0 / Ring 3）是什么，硬件如何实现
- 为什么需要两种模式（不是设计偏好，是安全必要）
- 从用户态进入内核态的三条路：syscall、中断、异常
- 什么是上下文切换（context switch），代价有多大
- 为什么频繁的内核态切换是沙箱设计的核心性能瓶颈

## 1. CPU 特权级：硬件里的"权限系统"

### 类比：公司的权限级别

> 💡 **类比**：想象一家公司有两种员工：
> - **普通员工（Ring 3）**：只能访问自己的工作电脑、自己的文件夹，不能动服务器机房。
> - **系统管理员（Ring 0）**：可以进机房、重启任何服务器、看任何人的文件。
>
> CPU 的特权级是这个概念的硬件实现。

**特权级**（Privilege Level），x86 架构称之为**保护环**（Protection Ring），是 CPU 硬件实现的权限机制。x86-64 定义了 Ring 0 到 Ring 3 共四个级别，但 Linux 只使用两个：

```
┌─────────────────────────────────────────────────────────────────┐
│                     x86-64 特权级（Linux 视角）                  │
│                                                                  │
│   ┌──────────────────────────────────────────────────────────┐  │
│   │                   Ring 0 — 内核态                        │  │
│   │                                                          │  │
│   │  权限：                                                  │  │
│   │  ✓ 执行所有 CPU 指令（包括特权指令）                     │  │
│   │  ✓ 访问所有物理内存地址                                  │  │
│   │  ✓ 读写所有 I/O 端口（直接操作硬件）                    │  │
│   │  ✓ 修改 CR3 寄存器（切换页表，即切换进程地址空间）       │  │
│   │  ✓ 修改中断向量表                                       │  │
│   │  ✓ 关闭/开启中断（CLI/STI 指令）                        │  │
│   │                                                          │  │
│   │  谁在这里：Linux 内核                                    │  │
│   └──────────────────────────────────────────────────────────┘  │
│                      ▲ 只能通过受控机制进入 ▼                    │
│   ┌──────────────────────────────────────────────────────────┐  │
│   │                   Ring 3 — 用户态                        │  │
│   │                                                          │  │
│   │  限制：                                                  │  │
│   │  ✗ 不能执行特权指令（执行 → 触发 #GP 异常 → 进程被杀）  │  │
│   │  ✗ 不能访问内核空间内存（访问 → 页错误 → SIGSEGV）      │  │
│   │  ✗ 不能直接访问硬件 I/O 端口                            │  │
│   │  ✗ 不能修改 CR3（不能给自己换页表）                     │  │
│   │                                                          │  │
│   │  谁在这里：你的所有程序（浏览器、Python、gcc、bash…）    │  │
│   └──────────────────────────────────────────────────────────┘  │
│                                                                  │
│   (Ring 1, Ring 2 存在于 x86 设计中，Linux 不使用它们)           │
└─────────────────────────────────────────────────────────────────┘
```

CPU 当前运行在哪个特权级，由一个叫 **CS 寄存器**（Code Segment Register）中的两个 bit（CPL，Current Privilege Level）决定。用户态时 CPL=3，内核态时 CPL=0。

### 为什么不能"伪装"成内核

你可能想问：我能不能把 CS 寄存器里的 CPL 改成 0，让自己变成内核态？

**不行。** 修改 CS 的方式是执行段切换指令（如 `far call`），而切换到更高特权级必须经过预先注册的**门**（gate descriptor）——这些门是内核在系统初始化时设置的，只有内核自己能设置。用户态代码没有任何办法直接将自己提升到 Ring 0。

## 2. 为什么需要两种模式

历史上有过没有特权级保护的操作系统（比如早期 MS-DOS）。在那个世界：

- 任何程序都可以读写内核数据结构，一个 bug 可以让整个系统崩溃
- 任何程序都可以直接操作硬件，一个恶意程序可以格式化硬盘
- 任何程序都可以读取其他程序的内存，密码毫无保护可言

两种模式解决三个根本问题：

| 问题 | 没有特权级 | 有特权级 |
|------|-----------|---------|
| **内核保护** | 用户程序可以覆盖内核代码 | 内核空间用户态不可访问 |
| **硬件隔离** | 用户程序可以直接驱动硬件 | 硬件操作只能通过内核 API |
| **进程隔离** | 进程可以读写彼此的内存 | MMU 强制执行页表隔离 |

## 3. 进入内核态的三条路

用户态代码在三种情况下会进入内核态：

### 路线 A：系统调用（Syscall）——主动请求

这是**主动的、有意的**内核态进入。程序需要内核服务时发起。

```
用户程序：                    内核：

  …代码…                      等待…
  RAX = 1  (write 的编号)
  RDI = 1  (fd=stdout)
  RSI = msg_ptr
  RDX = 6  (长度)
  SYSCALL 指令 ────────────►  保存用户态寄存器
                              跳转到 sys_write 处理函数
                              验证参数
                              执行写操作
  ◄────────────────────────── 恢复用户态寄存器
  SYSRET 指令                 返回值放入 RAX
  从 SYSCALL 后继续执行
```

> 💡 **SYSCALL 指令**是 x86-64 引入的专用系统调用指令，比旧的 `int 0x80` 方式快约 3 倍。执行 SYSCALL 时，CPU 自动：保存 RIP 和 RFLAGS → 从 MSR（Model Specific Register）加载内核入口点地址 → 将 CPL 设置为 0。

### 路线 B：硬件中断（Hardware Interrupt）——外部触发

这是**被动的、异步的**内核态进入。来自外部硬件设备。

```
时间线：
─────────────────────────────────────────────────────►

用户程序运行…  ←── 网卡收到数据包 ──→ 中断！
     │                                     │
     │  CPU 暂停用户程序                   │
     │  保存当前所有寄存器到内核栈         │
     │  切换到 Ring 0                      │
     │  跳转到对应的中断处理程序           │
     │                                     │
     │  内核处理：把数据复制到内核缓冲区   │
     │  内核处理：唤醒等待数据的进程       │
     │                                     │
     ▼  恢复寄存器，IRET 返回用户态        │
用户程序继续执行（完全不知道刚才发生了什么）
```

**中断向量表**（IVT / IDT）是内核维护的一个数组，共 256 个条目，每个条目是一个中断处理程序的地址。键盘按键、鼠标移动、网络数据到达、定时器滴答……每种硬件事件都有对应的中断号。

### 路线 C：异常（Exception）——指令执行错误

这是**同步的**内核态进入，因为 CPU 执行指令时遇到了问题。

| 异常类型 | 触发原因 | 内核的典型处理 |
|---------|---------|--------------|
| **页错误**（#PF, Page Fault） | 访问未映射/权限错误的地址 | 分配物理页 或 发送 SIGSEGV |
| **一般保护异常**（#GP） | 执行特权指令 / 访问越界 | 发送 SIGSEGV |
| **除零异常**（#DE） | 整数除以零 | 发送 SIGFPE |
| **无效操作码**（#UD） | 执行无效/未知指令 | 发送 SIGILL |
| **断点**（#BP） | 执行 `INT3` 指令 | 通知 ptrace 调试器 / 发送 SIGTRAP |

> 💡 **页错误（Page Fault）并不总是 bug**！当你访问一个还没分配物理内存的虚拟页时，内核会触发页错误处理程序，分配物理页，更新页表，然后重新执行那条指令——整个过程对用户程序**透明**。这是懒分配的实现机制。

## 4. 上下文切换：代价有多大

### 什么是上下文切换

**上下文切换**（Context Switch）是操作系统在多个进程之间分时共享 CPU 的机制。当内核决定从进程 A 切换到进程 B 时：

```
当前状态：CPU 在运行进程 A

1. 定时器中断触发（通常每 1-10ms）
   → 进入内核态（Ring 0）

2. 保存进程 A 的状态到其 task_struct：
   ├── 所有通用寄存器（RAX, RBX, RCX, …, R15）
   ├── RIP（下一条指令地址）
   ├── RSP（栈指针）
   ├── RFLAGS（标志寄存器）
   └── 浮点/SIMD 寄存器（如果使用了）

3. 调度器选择下一个要运行的进程 B

4. 切换到进程 B 的页表
   → 将 CR3 寄存器设置为进程 B 的页目录物理地址
   → TLB（地址翻译缓存）被全部或部分刷新

5. 恢复进程 B 的状态（从其 task_struct 加载寄存器）

6. 返回用户态，从进程 B 上次中断的地方继续执行
```

### 上下文切换的代价

上下文切换本身（保存/恢复寄存器）大约需要 **1-5 微秒**，听起来很快。但代价不止于此：

```
上下文切换的真实代价：

直接代价（快）：
  保存/恢复寄存器       ~1-3 μs
  切换页表（写 CR3）    ~0.1 μs

间接代价（慢，且难以预测）：
  TLB 刷新              每次缺失 ~50-200 个周期
  CPU 缓存（L1/L2/L3）失效  几 MB 的"热数据"消失
  分支预测器历史清空    几百个周期的预测错误

总代价（典型场景）：~1-10 μs 直接 + 几十到几百 μs 缓存重建
```

这就是为什么系统调用（每次至少 2 次特权级切换：进入 + 返回）虽然单次只需约 **100-300 纳秒**，但在高频场景下会成为瓶颈。

> 💡 **比较**：
> - 函数调用：~1 纳秒
> - 系统调用（ring 切换）：~100-300 纳秒（慢 100-300 倍！）
> - 进程间通信（IPC）：~1-10 微秒（涉及调度切换）

### 内核态切换频率的影响

每次程序与外界交互——读文件、写网络、创建进程——都要经过系统调用，也就是至少一次 Ring 3 → Ring 0 → Ring 3 的来回。

一个频繁 I/O 的程序（比如 web 服务器）每秒可能发起数十万次系统调用。这正是：
- **seccomp** 过滤每条 syscall 时要尽量快
- **ptrace** 沙箱（每条 syscall 都要进内核通知 tracer 进程）极慢的原因
- **shimmy** 要在用户态直接拦截、避免内核介入的根本动机

## 5. 特权级切换的硬件细节

SYSCALL 指令执行时，CPU 做了以下事情（硬件自动完成，不需要软件介入）：

```
SYSCALL 指令执行步骤（x86-64）：

1. 保存 RIP → RCX（用户态返回地址）
2. 保存 RFLAGS → R11
3. 从 IA32_LSTAR MSR 加载新 RIP（内核入口点）
4. 从 IA32_STAR MSR 加载新 CS/SS（Ring 0 段选择子）
   → CPL 变为 0（进入内核态）
5. 用 IA32_FMASK MSR 屏蔽部分 RFLAGS 标志位

此时 CPU 以 Ring 0 从内核入口点开始执行：
6. 内核保存剩余寄存器（RAX, RSI, RDI, R8-R10… RCX 和 R11 已被 CPU 覆盖）
7. 读取 RAX（系统调用号），在 syscall 表中查找处理函数
8. 调用处理函数（如 sys_write）
9. 处理函数将返回值放入 RAX
10. 内核恢复寄存器
11. 执行 SYSRET 指令：
    - 从 RCX 恢复 RIP
    - 从 R11 恢复 RFLAGS
    - 加载 Ring 3 的 CS/SS
    → CPL 变为 3（返回用户态）
12. 用户程序从 SYSCALL 后一条指令继续
```

这 12 步，每次系统调用都要走一遍。

## 小结

```
两种模式的本质：
    CPU 硬件保护 → Ring 0（内核）vs Ring 3（用户）

进入内核态的三条路：
    ① syscall 指令（主动，程序发起）
    ② 硬件中断（被动，设备触发）
    ③ 异常（被动，指令错误触发）

代价：
    每次 Ring 切换 ~100-300 ns
    + TLB/缓存污染（隐性代价）
    → 频繁切换是沙箱的性能瓶颈
```

**核心要点**：
1. Ring 0/Ring 3 是 CPU 硬件实现的，用户态程序无法绕过
2. 用户态程序只能通过三条受控路径进入内核
3. 每次内核态切换都有真实的性能代价（约 100-300 ns + 缓存影响）
4. 高频 syscall 拦截（如 ptrace 沙箱）正是因此变得极慢

## 与沙箱技术的联系

沙箱的核心工作是**拦截被沙箱程序的系统调用**——因为 syscall 是程序与外界交互的唯一通道，控制了 syscall 就控制了程序能做什么。

但拦截有代价：
- **ptrace 沙箱**：每条 syscall 都触发内核介入，通知 tracer 进程，这需要两次额外的上下文切换（tracer 和 tracee 之间）。代价极高，shimmy 的研究发现 ptrace 开销可高达原始性能的 10-100 倍。
- **seccomp-BPF**：在内核态直接过滤 syscall，避免了用户态 tracer 进程，代价约 100-300 ns/syscall。
- **shimmy（zpoline/lazypoline 方案）**：通过二进制插桩，在用户态拦截 syscall，完全绕过内核介入，代价仅约 10-50 ns/syscall。

这条优化链的本质，正是不断减少内核态↔用户态切换次数。
