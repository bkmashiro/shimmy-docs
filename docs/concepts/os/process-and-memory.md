# 进程与内存空间

> 你写的程序和"正在运行的程序"是两回事。本文深入解释进程是什么、它如何拥有自己的内存世界，以及操作系统如何在幕后维持这个精心设计的假象。

## 前置知识

- 知道什么是变量、函数调用
- 了解基本的 C 语法（能读懂 `int x = 5; malloc(100);`）
- 读过或了解"程序是什么"的基本概念

## 你将学到

- 进程与程序的本质区别
- 虚拟地址空间是什么，为什么每个进程都以为自己独占内存
- 页表和 MMU 如何完成地址翻译
- 进程内存的完整布局（代码段、数据段、BSS、堆、栈、mmap）
- 进程隔离为什么对安全至关重要

## 1. 程序 vs 进程

**程序**（program）是存储在磁盘上的静态文件——一堆字节，描述"要做什么"。就像食谱写在纸上，它本身不会做饭。

**进程**（process）是程序的**运行实例**——内核加载程序、分配资源、开始执行后，就产生了一个进程。同一个程序可以同时有多个进程在运行（比如你开了三个终端，就有三个 bash 进程）。

> 💡 **类比**：程序 = 乐谱；进程 = 乐队正在演奏。同一首曲子可以由多支乐队同时演奏，互不干扰。

进程包含什么：

```
进程 = 程序代码
     + 当前执行位置（RIP 寄存器）
     + 所有寄存器的当前值
     + 虚拟地址空间（它的内存世界）
     + 打开的文件列表
     + 信号处理器
     + 用户身份（uid/gid）
     + 其他元数据（PID、父进程、环境变量…）
```

内核为每个进程维护一个叫 **PCB**（Process Control Block，进程控制块）的数据结构，保存上面这些信息。Linux 里这个结构叫 `task_struct`，定义在内核源码中，有几百个字段。

## 2. 虚拟地址空间：每个进程的私人内存世界

### 问题：物理内存只有一份

你的机器可能有 16GB 物理内存。但同时运行着浏览器、IDE、终端、音乐播放器……它们加起来"需要"的内存可能远超 16GB。而且，如果进程 A 和进程 B 都把数据放在地址 `0x1000`，会不会互相覆盖？

### 解决方案：虚拟地址空间

操作系统给每个进程一个**私人的、连续的地址空间**，叫做**虚拟地址空间**（Virtual Address Space）。在 64 位 Linux 上，每个进程都以为自己有约 **128 TB** 的地址空间（从 `0x0000000000000000` 到 `0x00007FFFFFFFFFFF`）。

> 💡 **类比**：进程就像租了一套"虚拟公寓"。每个租户都以为自己有一栋独立的楼，可以把家具（数据）放在任何房间（地址）。但实际上，房东（内核）把所有租户的需求映射到真实的土地（物理内存）上。两个租户都可以说"我的卧室在 101 室"——因为他们的楼是不同的。

这个机制带来两个好处：

1. **隔离**：进程 A 的地址 `0x1000` 和进程 B 的地址 `0x1000` 对应不同的物理内存，互不干扰。
2. **超额分配**：多个进程可以"声称"大量虚拟内存，只有真正使用时才分配物理内存（懒分配）。

## 3. 页表与 MMU：地址翻译的机器

### 页（Page）

内存不是按字节管理的，而是按**页**（Page）管理的。x86-64 Linux 默认页大小是 **4 KB**（4096 字节）。

虚拟地址空间被分成一个个 4KB 的虚拟页，物理内存也被分成 4KB 的物理帧（physical frame）。

### 页表（Page Table）

**页表**是内核为每个进程维护的映射表，记录每个虚拟页对应哪个物理帧。

```
进程 A 的页表示意：

虚拟页号   物理帧号   标志位
─────────────────────────────────
VPN 0x400  → PFN 0x1A0  | 可读可执行
VPN 0x401  → PFN 0x2B1  | 可读可执行
VPN 0x600  → PFN 0x05C  | 可读可写
VPN 0x700  → (未映射)   | —— (访问→页错误)
```

在 x86-64 上，实际的页表是四层结构（PML4 → PDPT → PD → PT），但概念是相同的：虚拟地址 → 物理地址。

### MMU：地址翻译硬件

**MMU**（Memory Management Unit，内存管理单元）是 CPU 内部的一个硬件模块，负责**自动、透明地**完成虚拟地址到物理地址的翻译。

```
地址翻译流程：

                CPU 执行指令
                访问虚拟地址 0x401008
                      │
                      ▼
              ┌───────────────┐
              │     MMU       │
              │               │
              │  查页表：      │
              │  VPN 0x401    │
              │  → PFN 0x2B1  │
              │               │
              │  物理地址 =    │
              │  0x2B1000     │
              │  + 偏移 0x008 │
              │  = 0x2B1008   │
              └───────┬───────┘
                      │
                      ▼
              访问物理内存 0x2B1008
```

> 💡 **类比**：虚拟地址就像一个"内部工号"，MMU 是前台，手里有一本"工号→真实姓名"的花名册（页表）。每次有人用内部工号找人，前台自动查表转换成真实姓名。

当切换到另一个进程时，内核告诉 MMU 切换到那个进程的页表——于是同样的虚拟地址就会翻译到不同的物理地址，隔离完美实现。

## 4. 内存布局：进程虚拟空间里住着什么

一个典型的 64 位 Linux 进程的虚拟地址空间布局如下：

```
虚拟地址（从高到低）

0xFFFFFFFFFFFFFFFF ┌──────────────────────────────────────┐
                   │  内核空间（进程不可见）                │
                   │  内核代码、数据、页表本身都在这里      │
0xFFFF800000000000 ├──────────────────────────────────────┤
                   │                                      │
                   │  ~~~ 不可访问的空洞 ~~~               │
                   │  (访问任何此处地址 → SIGSEGV)         │
                   │                                      │
0x00007FFFFFFFFFFF ├──────────────────────────────────────┤
                   │  栈（Stack）                          │ ← 向低地址增长 ↓
                   │  函数局部变量、参数、返回地址          │
                   │  [ASLR 随机化起始位置]                │
                   ├──────────────────────────────────────┤
                   │         ↕ 增长空间                    │
                   ├──────────────────────────────────────┤
                   │  mmap 区域                            │
                   │  动态库（libc.so 等）                  │
                   │  匿名映射（大 malloc 用此）            │
                   │  文件映射                             │
                   ├──────────────────────────────────────┤
                   │         ↕ 增长空间                    │
                   ├──────────────────────────────────────┤
                   │  堆（Heap）                           │ ← 向高地址增长 ↑
                   │  malloc/new 分配的内存                │
                   ├──────────────────────────────────────┤
                   │  .bss 段                              │
                   │  未初始化全局变量（全为 0）            │
                   ├──────────────────────────────────────┤
                   │  .data 段                             │
                   │  已初始化全局变量                      │
                   ├──────────────────────────────────────┤
                   │  .rodata 段                           │
                   │  只读数据（字符串常量等）              │
                   ├──────────────────────────────────────┤
0x0000000000401000 │  .text 段（代码段）                   │
                   │  程序的机器码指令                      │
                   │  CPU 的 RIP 在这里跳来跳去             │
0x0000000000400000 ├──────────────────────────────────────┤
                   │  ELF 程序头                           │
0x0000000000000000 └──────────────────────────────────────┘
                   NULL（访问此处 → SIGSEGV）
```

### 各区域详解

**代码段（.text）**
- 存放程序的机器码指令
- 权限：只读 + 可执行（r-x）
- CPU 的 RIP 指针在这里跳来跳去
- 为什么只读？防止程序在运行时修改自己的代码（自修改代码是安全漏洞的温床）

**只读数据段（.rodata）**
- 字符串常量（`"Hello, World!"`）
- `const` 全局变量
- 权限：只读（r--）

**已初始化数据段（.data）**
- 有初始值的全局变量：`int x = 5;`
- 权限：可读可写（rw-）
- 初始值存储在 ELF 文件中，加载时复制到内存

**BSS 段（.bss）**
> 💡 **BSS 的名字来历**：Block Started by Symbol，是汇编时代的遗留名称，现在意思是"未初始化的全局变量"。

- 没有初始值的全局变量：`int arr[1000];`
- 权限：可读可写（rw-）
- 在 ELF 文件里**不占空间**——只记录"这里需要多少字节，全填 0"
- 加载时由内核负责清零

**堆（Heap）**
- `malloc()`、`calloc()`、C++ `new` 分配的内存
- 由 `brk`/`sbrk` 系统调用扩张（小块分配）或 `mmap` 匿名映射（大块分配）
- 向高地址增长
- 需要手动管理（C/C++），或由 GC 自动管理（Python/Java）

**mmap 区域**
- 动态链接库（libc.so、libpthread.so 等）加载到这里
- `mmap()` 系统调用映射的文件或匿名内存
- 大块 `malloc`（通常 > 128KB）也在这里

**栈（Stack）**
- 函数调用时自动管理
- 每次调用函数：压入一个**栈帧**（stack frame），包含局部变量、保存的寄存器、返回地址
- 函数返回：弹出栈帧
- RSP 寄存器始终指向栈顶
- 向低地址增长（栈越用越大，地址越来越小）
- 默认大小通常 8MB（可用 `ulimit -s` 查看）

```
函数调用时栈的变化：

调用前                调用 foo(3, 4) 后

高地址               高地址
┌───────────┐        ┌───────────┐
│           │        │           │
│  main 的  │        │  main 的  │
│  栈帧     │        │  栈帧     │
│  (局部变量)│        │  (局部变量)│
│  返回地址  │  RSP→  │  返回地址  │
└───────────┘        ├───────────┤
                     │  foo 的   │ ← RSP 压低
                     │  参数     │
                     │  局部变量  │
              RSP→   │  ...      │
低地址               低地址
```

## 5. 实际验证：用 /proc 看内存布局

Linux 把每个进程的内存映射暴露在 `/proc/<PID>/maps` 文件中。

```bash
$ cat /proc/self/maps
555555554000-555555555000 r--p 00000000 08:01 1234567  /bin/cat
555555555000-555555556000 r-xp 00001000 08:01 1234567  /bin/cat
555555556000-555555557000 r--p 00002000 08:01 1234567  /bin/cat
555555558000-555555559000 r--p 00003000 08:01 1234567  /bin/cat
555555559000-55555555a000 rw-p 00004000 08:01 1234567  /bin/cat
7ffff7dc5000-7ffff7dea000 r--p 00000000 08:01 2345678  /lib/x86_64-linux-gnu/libc.so.6
7ffff7dea000-7ffff7f6a000 r-xp 00025000 08:01 2345678  /lib/x86_64-linux-gnu/libc.so.6
...
7ffffffde000-7ffffffff000 rw-p 00000000 00:00 0        [stack]
```

每行格式：`起始地址-结束地址  权限  文件偏移  设备  inode  路径`

权限字母：`r`（可读）`w`（可写）`x`（可执行）`p`（私有，写时复制）

## 6. 为什么隔离很重要

### 崩溃隔离

进程 A 的程序 bug 导致访问无效地址：

```
进程 A 代码: *ptr = 42;   // ptr 是 NULL！

1. CPU 执行这条指令，MMU 查页表
2. 虚拟地址 0x0 没有映射 → 页错误（Page Fault）
3. 内核接管，判断这是非法访问
4. 内核向进程 A 发送 SIGSEGV 信号
5. 进程 A 崩溃、终止
6. 进程 B、C、D 完全不受影响，继续正常运行
```

没有虚拟内存保护，进程 A 的 bug 可能覆盖进程 B 的数据，导致连锁崩溃，整个系统瘫痪。

### 安全隔离

进程不能读取其他进程的内存。即使进程 B 知道进程 A 的某个秘密（比如密码）存储在虚拟地址 `0x7fff0000`，它也无法访问——因为这个地址对进程 B 的页表来说要么没映射，要么映射到完全不同的物理内存。

> 💡 **沙箱视角**：虚拟内存隔离是操作系统级沙箱的基础。但进程内（同一进程中的不同代码，比如插件和宿主程序）就没有这种硬件级隔离，这正是 shimmy 需要在进程内提供细粒度保护的原因。

## 小结

```
程序（磁盘上的文件）
    ↓ 内核加载、分配资源
进程（运行实例）
    │
    ├── 虚拟地址空间（每进程独享的假象）
    │       ├── .text（代码，r-x）
    │       ├── .rodata（只读数据，r--）
    │       ├── .data（初始化全局变量，rw-）
    │       ├── .bss（零初始化全局变量，rw-）
    │       ├── 堆（malloc，向上增长）
    │       ├── mmap（动态库、大内存）
    │       └── 栈（函数调用，向下增长）
    │
    └── 页表（虚拟→物理地址映射，由 MMU 硬件执行）
```

**核心要点**：
1. 进程是程序的运行实例，包含运行时所有状态
2. 每个进程有私有的虚拟地址空间，假装独占全部内存
3. 页表 + MMU 在硬件层面完成地址翻译和隔离
4. 内存布局有严格的区域划分，每个区域权限不同
5. 一个进程崩溃不会影响其他进程，因为物理内存是隔离的

## 与沙箱技术的联系

虚拟内存隔离是 **进程级沙箱** 的天然基础：Docker、虚拟机等容器技术都依赖进程拥有独立地址空间这一事实。

但 shimmy 面对的挑战更复杂：沙箱代码（被测程序）和宿主代码（DynamoRIO 框架）运行在**同一个进程**里，共享同一个地址空间。这意味着操作系统的进程隔离对它们不适用，shimmy 必须通过软件手段（syscall 拦截、内存权限控制）在进程内部实现隔离。`mprotect` 系统调用正是用来修改特定内存页的权限，配合 MPK（Memory Protection Keys）可以在不切换进程的情况下实现细粒度隔离——这些都是 shimmy 核心设计的组成部分。
