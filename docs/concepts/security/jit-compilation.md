# JIT 编译

**前置知识**：了解"解释器"和"编译器"的区别（指南第一章）；知道"机器码"是什么；了解基本的内存保护概念（可执行 vs 不可执行）有帮助。

**你将学到**：

- JIT 的核心思路：热点代码识别与即时编译
- JIT 为什么需要 W⊕X（写和执行互斥）
- JIT spraying 攻击：攻击者如何利用 JIT 生成 shellcode
- 在沙箱中做 JIT 的挑战

## 1. 三种执行方式

想象你是一名外国游客，拿到了一份中文菜单，想点菜。你有三种策略：

1. **解释执行**：现查词典，看一个字查一个字，每次点菜都要查（慢，但灵活）
2. **编译执行**：出发前把整份菜单全部翻译好带着（出发前慢，点菜时快）
3. **JIT 编译**：先用第一种方法，同时记下经常点的菜，第二次再点同样的菜时直接从笔记里找（第一次慢，之后快）

> 💡 **JIT 编译（Just-In-Time Compilation）**
> "即时编译"。程序先以解释方式运行，运行时记录哪些代码执行得最频繁（"热点代码"），然后把这些热点代码编译成机器码，下次执行时直接运行机器码，跳过解释步骤。

### 1.1 纯解释执行

```python
# Python 代码
for i in range(1000000):
    x = i * i + 1

# CPython（Python 解释器）的执行过程：
# 每次循环：
#   1. 取下一条字节码指令
#   2. switch-case 查找对应的 C 函数
#   3. 执行 C 函数
#   4. 回到步骤 1
# 100 万次 × 每次的解释开销 = 很慢
```

**优点**：启动快，代码改动立即生效，内存用量小
**缺点**：热代码反复被解释，大量时间浪费在"理解"同一段代码上

### 1.2 提前编译（AOT，Ahead-of-Time）

```
源代码 ──(编译器)──► 机器码 ──► 直接执行
         慢（分钟级）           快（接近硬件极限）
```

C/C++/Rust 的工作方式。

**优点**：运行速度最快，编译器有充足时间做深度优化
**缺点**：每次改代码都要重新编译；无法根据运行时信息优化（比如实际输入的数据类型）

### 1.3 JIT 编译

```
解释执行
    +
运行时性能分析（Profiling）
    +
热点代码即时编译
    =
兼顾启动速度和运行速度
```

```
JIT 执行流程：

程序开始运行
    │
    ▼
解释执行字节码（或 IR）
    │
    ├── 同时记录执行计数
    │   if 函数调用次数 > 阈值（如 10000 次）:
    │       标记为"热点"，触发 JIT 编译
    ▼
JIT 编译器（后台线程）：
  1. 把热点函数的字节码 → 机器码
  2. 机器码放入代码缓存（可执行内存）
  3. 修改"调用表"，下次调用直接执行机器码
    │
    ▼
热点函数：直接执行机器码（接近原生速度）
非热点函数：继续解释执行
```

**使用 JIT 的典型系统**：
- V8（Chrome/Node.js 的 JavaScript 引擎）
- JVM（Java 虚拟机）HotSpot
- PyPy（Python 的高性能替代实现）
- LuaJIT
- WASM 运行时（Wasmtime、V8 的 WASM 支持）

## 2. JIT 的实现：生成和执行机器码

JIT 编译器需要在运行时生成机器码并执行，这涉及一个根本性的安全问题。

### 2.1 JIT 需要可执行的可写内存

普通程序的内存权限：

```
内存区段         权限
─────────────────────────────────
.text（代码段）    r-x（可读、可执行，不可写）
.data（数据段）    rw-（可读写，不可执行）
栈               rw-（通常不可执行）
堆               rw-（通常不可执行）
```

这种分隔是安全的：代码不会被意外修改，数据不会被意外执行。

JIT 编译器需要打破这种分隔：它需要在运行时**写入**新生成的机器码，然后**执行**这段代码。

### 2.2 W⊕X 原则

> 💡 **W⊕X（Write XOR Execute，写与执行互斥）**
> 一种安全原则：同一块内存在同一时刻要么可写（W），要么可执行（X），但不能同时两者兼具。这阻止了"把数据当代码执行"的攻击。

在 Linux 上，通过 `mmap` 和 `mprotect` 实现 W⊕X：

```c
// JIT 编译器的正确做法：先写，再改权限，然后执行

// 第一步：分配可写（但不可执行）的内存
uint8_t *code_buf = mmap(NULL, page_size,
                          PROT_READ | PROT_WRITE,   // 可读写
                          MAP_PRIVATE | MAP_ANONYMOUS,
                          -1, 0);

// 第二步：写入生成的机器码
// 此时内存可写，可以安全写入
memcpy(code_buf, generated_machine_code, code_size);

// 第三步：改变内存权限为可执行（同时移除写权限）
mprotect(code_buf, page_size,
         PROT_READ | PROT_EXEC);   // 可读可执行，但不可写！

// 第四步：执行生成的代码
typedef void (*func_t)(void);
func_t func = (func_t)code_buf;
func();  // 现在执行这段代码是安全的

// 如果需要修改代码：
// 1. 先 mprotect 改回 PROT_READ | PROT_WRITE
// 2. 写入新代码
// 3. 再 mprotect 改为 PROT_READ | PROT_EXEC
// → 确保"写"和"执行"永远不同时有效
```

W⊕X 的意义：即使攻击者能向代码缓冲区写入数据（比如通过缓冲区溢出），由于此时内存不可执行，这段数据也不能被当作代码运行。

### 2.3 Apple Silicon 上的 W⊕X

苹果的 M 系列芯片（ARM64）强制执行 W⊕X——硬件层面不允许同一块内存同时可写且可执行。JIT 编译器必须使用特殊的线程本地权限临时切换机制（`MAP_JIT` flag + `pthread_jit_write_protect_np`）。

## 3. JIT Spraying 攻击

JIT spraying（JIT 喷洒）是一类利用 JIT 编译器生成 shellcode 的攻击技术，由 Dion Blazakis 在 2010 年提出。

### 3.1 背景：为什么 NX 位不够

**NX（No-Execute）位**（又叫 DEP，数据执行预防）是 W⊕X 的硬件实现：标记数据段为不可执行，防止攻击者把 shellcode 注入数据段然后跳过去执行。

但是 JIT 引擎有合法的理由创建可执行内存……

### 3.2 攻击原理

JIT spraying 的思路是：**让 JIT 引擎帮攻击者"生成"shellcode**。

```
攻击步骤（以 JavaScript + V8 为例）：

1. 攻击者构造特殊的 JavaScript 代码：

   var a = (0x90909090 + 0x90909090);  // 0x90 = NOP 指令
   var b = (0x90909090 + 0x90909090);
   var c = (0x90909090 + 0x90909090);
   // 大量重复...

2. V8 的 JIT 编译器编译这些加法操作，
   生成的机器码中会包含大量常量 0x90909090：

   mov rax, 0x90909090  ; 操作数编码在指令里
   add rax, 0x90909090
   mov rbx, 0x90909090
   ...

3. 这些 0x90 字节在机器码中大量出现，
   而 0x90 正是 x86 的 NOP 指令！

4. 如果攻击者再有另一个漏洞（比如控制流劫持），
   让 RIP 跳到 JIT 代码区域的某个位置，
   那里会有大量 NOP sled（NOP 滑行段），
   最终执行到攻击者精心嵌入的 shellcode。
```

```
JIT 生成的内存（简化示意）：

[合法的 JIT 代码] [0x90 0x90 0x90...] [攻击者的 shellcode 字节被编码进常量里] [更多代码]
                  ↑
         攻击者通过另一个漏洞跳到这里
         NOP sled 保证跳到 shellcode
```

### 3.3 防御措施

**ASLR（地址空间布局随机化）**：把 JIT 代码放到随机地址，攻击者不知道 JIT 代码在哪里。但如果有信息泄露漏洞（info leak），ASLR 可以被绕过。

**常量混淆（Constant Blinding）**：JIT 编译器在生成包含常量的指令前，先对常量 XOR 一个随机密钥，执行时再 XOR 还原：

```asm
; 原本的 JIT 代码（可能被 spraying 利用）：
mov rax, 0x90909090

; 常量混淆后：
mov rax, 0x90909090 XOR key  ; key 是随机的，攻击者不知道
xor rax, key                  ; 执行时还原
; 这样常量字节随机化，无法预测 JIT 代码里有哪些字节
```

**CFI（控制流完整性）**：确保间接跳转只能跳到合法的目标地址，即使攻击者控制了 RIP，也只能跳到预先授权的位置。

## 4. 在沙箱里做 JIT 的挑战

### 4.1 沙箱需要 PROT_EXEC，但 JIT 也需要

在沙箱场景下，我们通常想禁止程序动态生成可执行代码（因为这可能被用来注入 shellcode）。但是：

- Python（PyPy）需要 JIT
- JavaScript（V8、SpiderMonkey）需要 JIT
- Java（JVM HotSpot）需要 JIT
- WASM 运行时（Wasmtime、WasmEdge）需要 JIT

如果沙箱完全禁止 `mmap(PROT_EXEC)` 和 `mprotect(PROT_EXEC)`，这些运行时就无法工作（或者只能以慢得多的解释模式运行）。

```
沙箱的两难：
  禁止 PROT_EXEC → JIT 语言无法运行（或极慢）
  允许 PROT_EXEC → 程序可以动态生成任意机器码
                   → 可能绕过沙箱！
```

### 4.2 JIT 代码绕过系统调用拦截

如果沙箱通过修改代码来拦截 `syscall` 指令（比如 DynamoRIO 的方式），JIT 代码是个大问题：

```
问题流程：
  1. 沙箱扫描程序的初始代码，插桩所有 syscall 指令 ✓
  2. JIT 编译器在运行时动态生成新的机器码
  3. 新代码里可能包含 syscall 指令
  4. 沙箱还没有扫描这段新代码！
  5. 攻击者可以通过 JIT 生成包含 syscall 指令的代码
  6. → 绕过沙箱

解决方案（DynamoRIO 的做法）：
  - 监控 mprotect/mmap 系统调用
  - 每当有新内存变为可执行，立即重新扫描并插桩
  - 对"自修改代码"做特殊处理（代码页被写入后清除缓存）
  - 这带来额外开销，但保证覆盖
```

### 4.3 Shimmy 的处理方式

Shimmy Sandbox 在处理 JIT 代码时采用的策略：

```
1. 拦截 mmap(PROT_EXEC) 调用
   - 记录新的可执行内存区域
   - 通知 DynamoRIO 扫描这些区域

2. 拦截 mprotect(PROT_EXEC) 调用
   - 类似上面

3. 在代码缓存失效时重新插桩
   - 检测到代码页被修改 → 清除对应的 trace
   - 下次执行时重新翻译并插桩

4. 对 io_uring 特别处理（它绕过了正常的 syscall 路径）
   - 拦截 io_uring_setup、io_uring_enter 等调用
   - 验证提交到环形缓冲区的操作的合法性
```

## 5. JIT 的性能收益与代价

### 5.1 V8 的 JIT 优化效果

```
JavaScript 执行性能（相对于纯解释，数值越高越快）：

纯解释（无 JIT）                 1×
Ignition（字节码解释器）          5×
SparkPlug（简单 JIT）            15×
Maglev（中层 JIT，2023 新增）    30×
Turbofan（顶层优化 JIT）         50×

注：实际数值因代码类型而异
```

现代 JavaScript 引擎的 JIT 速度已经接近 C++ 代码，这在 1995 年（JavaScript 发明之年）是不可想象的。

### 5.2 JIT 的冷启动问题

JIT 的性能只在代码多次执行后才体现。对于短暂的函数（只调用一两次），JIT 本身的开销可能比收益还大：

```
JIT 编译一个函数的成本：
  ~1000 微秒（复杂优化 JIT）
  ~10 微秒（简单 JIT）

函数执行节省的时间：
  如果原本每次执行需要 1 微秒（解释），JIT 后 0.1 微秒
  节省：0.9 微秒 / 次

→ 简单 JIT 需要至少 11 次调用才能回本
→ 复杂优化 JIT 需要至少 1111 次调用才能回本
```

这就是 V8 使用多层 JIT 的原因：用便宜的 SparkPlug 快速生成还不错的代码，用昂贵的 Turbofan 深度优化真正的热点。

## 小结

```
三种执行方式：
  解释执行  → 慢，但启动快，灵活
  AOT 编译  → 快，但编译慢，无法动态适应
  JIT 编译  → 兼顾启动和运行速度

W⊕X 原则：
  写和执行不能同时有效
  JIT 的正确做法：写完代码 → mprotect 移除写权限 → 执行

JIT spraying：
  利用 JIT 引擎生成含 shellcode 字节的可执行内存
  防御：ASLR + 常量混淆 + CFI

沙箱中的 JIT 挑战：
  需要 PROT_EXEC（沙箱想限制）
  动态生成代码绕过静态插桩
  解决：监控 mprotect/mmap，动态重插桩
```

## 与 Shimmy/沙箱设计的联系

JIT 编译对 Shimmy Sandbox 有直接影响：

1. **Python 的 JIT 支持**：如果用户运行 PyPy（JIT Python 实现）或包含 JIT 的 JavaScript 代码，Shimmy 需要允许 `mprotect(PROT_EXEC)`，同时确保新生成的代码也被正确插桩。

2. **DynamoRIO 本身就是 JIT**：DynamoRIO 将程序的原始代码 JIT 编译到自己的代码缓存中。这意味着 Shimmy 在用户程序的 JIT 上再叠了一层 JIT——需要特别小心两者的交互。

3. **`mmap(PROT_EXEC)` 的策略**：Shimmy 必须允许这个调用（否则 JIT 运行时无法工作），但同时要监控所有新产生的可执行内存并确保覆盖到拦截逻辑。

4. **`io_uring` 是个特殊的"JIT 绕过"**：`io_uring` 允许程序把系统调用放入环形缓冲区，内核异步执行，完全绕过 `syscall` 指令路径。Shimmy 必须特别处理这个接口，就像处理 JIT 生成的 syscall 指令一样。
