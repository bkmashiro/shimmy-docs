## 1. WebAssembly 是什么

**WebAssembly**（简称 WASM）是一种可移植的低级字节码格式。它设计的初衷是让 C/C++/Rust 等编译型语言编写的程序能在浏览器里安全运行，但现在它的应用远超浏览器范畴。

> 💡 **WebAssembly（WASM）**
> 一种紧凑的二进制指令格式，设计目标是：可移植（在任何平台的运行时上运行）、安全（天然沙箱化）、高效（接近原生性能）。就像 Java 字节码，但更接近底层，不与 Java 语言绑定。

### 1.1 WASM 的位置

```
C/C++/Rust 源码
       │
       │ clang/rustc + wasm32 target
       ▼
  .wasm 文件（二进制格式）
       │
       │ WASM 运行时（Wasmtime / WasmEdge / V8）
       ▼
  实际执行（JIT 或 AOT 编译到原生机器码）
```

你可以把 WASM 理解为一种"通用翻译格式"：不同语言的编译器都可以把代码翻译成 WASM，而不同的运行时都可以执行 WASM。

### 1.2 为什么浏览器需要 WASM

在 WASM 出现之前，浏览器只能执行 JavaScript。如果你想在浏览器里跑一个 C 写的图像处理库，你要么重写成 JS（慢），要么用 asm.js（性能还好，但是个临时方案）。

WASM 让 C/C++/Rust 代码可以直接在浏览器里运行，性能接近原生，同时保持安全隔离——网页里的 WASM 代码无法访问你的文件系统，无法发起任意网络请求，无法调用操作系统 API。

## 2. WASM 的安全模型

WASM 的安全性来自它的**设计约束**，而不是运行时的检查——它的指令集本身就不允许某些操作。

### 2.1 线性内存（Linear Memory）

WASM 程序的所有数据都存放在一块**线性内存**中——一个连续的字节数组，从地址 0 开始，大小可以增长但不能随意定位。

```
WASM 线性内存布局：

地址 0                                              最大地址
│                                                          │
▼                                                          ▼
┌────────────────────────────────────────────────────────┐
│  栈区  │  堆区  │  全局变量  │  代码数据  │  未使用区   │
│ (高地址)│       │            │            │             │
└────────────────────────────────────────────────────────┘
         ▲
   wasm 程序只能访问这块区域内的地址
   访问越界 → 运行时抛出 trap（类似 SIGSEGV，但受控）
```

关键特性：
- WASM 程序**只能访问**线性内存里的地址
- 线性内存和宿主机的内存是**完全隔离**的
- 访问越界会被运行时检测到并终止（trap），不会造成任意内存读写

这和 C 程序截然不同——C 程序可以把任意整数当指针来解引用，没有边界检查。

### 2.2 无裸指针（No Raw Pointers）

WASM 的指令集里没有"把任意整数作为内存地址"这种操作。

```
C 代码（危险）：
  int *p = (int *)0x12345678;  // 把任意整数作为指针
  *p = 42;                      // 可以写入任意内存地址！
  // 这可以用来改写内核数据结构、其他进程的内存...

WASM（安全）：
  WASM 中的"指针"实际上是线性内存内的**偏移量**
  i32.load offset=0x1000  // 读取线性内存第 0x1000 字节
  // 如果 0x1000 超出线性内存大小 → trap，不会真正访问任意内存
```

WASM 里的所有内存访问都相对于线性内存的基地址。运行时在执行 `i32.load` / `i32.store` 指令时，会检查偏移量是否在合法范围内。

### 2.3 类型化指令集

WASM 是**强类型**的——每个值都有明确的类型（`i32`、`i64`、`f32`、`f64`），指令只接受特定类型的操作数。

```
WASM 汇编（文本格式 .wat）示例：

(module
  (func $add (param $a i32) (param $b i32) (result i32)
    local.get $a   ;; 把 $a 压栈（i32 类型）
    local.get $b   ;; 把 $b 压栈（i32 类型）
    i32.add        ;; 弹出两个 i32，压入它们的和
  )
  (export "add" (func $add))
)
```

类型系统保证了：
- 不能把一个函数指针当整数使用
- 不能把浮点数当内存地址
- 所有函数调用都有类型检查

### 2.4 间接调用表（Indirect Call Table）

WASM 没有函数指针，只有**函数索引**。间接调用必须通过一个受控的函数表：

```
WASM 函数表：
  表的索引: [ 0 ] → 函数 A
            [ 1 ] → 函数 B
            [ 2 ] → 函数 C
            ...

间接调用：
  call_indirect (type $signature) (i32.const 2)
    // 调用索引为 2 的函数（函数 C）
    // 运行时检查：
    //   1. 索引 2 在表范围内？✓
    //   2. 函数 C 的类型与 $signature 匹配？✓
    //   → 允许调用
```

这防止了**控制流劫持攻击**：攻击者无法通过覆盖函数指针来让程序跳转到任意地址。

## 3. WASM 如何实现隔离

### 3.1 内存边界检查

每次内存访问，运行时都做：

```
WASM 执行 i32.load (offset=addr)：

1. 计算有效地址：eff_addr = wasm_linear_memory_base + addr
2. 检查范围：addr + 4 ≤ wasm_memory_size？
   ├── 否 → trap（运行时抛出异常）
   └── 是 → 执行 *(eff_addr)，返回值

                  宿主机物理内存
                  ┌────────────────────────────────────────────┐
                  │ ... │ WASM 线性内存 │ 运行时数据 │ 内核 ... │
                  │     │◄────────────►│            │          │
                  │     │  只允许访问   │            │          │
                  │     │  这个范围     │            │          │
                  └────────────────────────────────────────────┘
                         ▲             ▲
                  base_addr     base_addr + mem_size
```

在 x86-64 上，运行时可以利用**内存保护页**（mprotect）来让硬件帮忙做边界检查，而不是每次都执行软件检查——这显著减少了开销。

### 3.2 能力（Capability）模型

WASM 程序默认没有任何能力：
- 没有文件系统访问
- 没有网络访问
- 没有系统调用

要让 WASM 程序访问这些资源，宿主机必须显式传入"能力"（通常是特定的接口函数）：

```
// 宿主机代码（Rust + Wasmtime 示例）
use wasmtime::*;

let engine = Engine::default();
let module = Module::from_file(&engine, "my_program.wasm")?;
let mut store = Store::new(&engine, ());

// 只传入允许的接口
let mut linker = Linker::new(&engine);

// 允许：向 stdout 写出
linker.func_wrap("env", "print", |s: &str| {
    println!("{}", s);
})?;

// 不传入 open_file、connect 等接口
// → WASM 程序完全无法访问文件和网络

let instance = linker.instantiate(&mut store, &module)?;
```

这就是 WASM 隔离的核心：**宿主机完全控制 WASM 程序能做什么**，通过能力的授予和拒绝来实现沙箱。

## 4. 主流 WASM 运行时

### 4.1 Wasmtime

由 Bytecode Alliance（Mozilla、Fastly 等）开发，用 Rust 编写。

```
特点：
  - 生产就绪，被 Fastly、Cloudflare 等用于生产环境
  - 支持 AOT（Ahead-of-Time）编译：预先编译为原生代码，启动极快
  - 严格的安全审计
  - WASI（WebAssembly System Interface）支持，提供标准化的系统接口

性能：与原生代码差距约 1.5×（AOT 模式）
```

### 4.2 WasmEdge

由 CNCF（云原生计算基金会）孵化，专注于边缘计算和云原生场景。

```
特点：
  - 启动时间极短（< 1ms），适合 serverless
  - 支持 WASI 和扩展接口
  - 提供 JavaScript 支持（QuickJS 嵌入）
  - 用于 Docker 的 WASM 支持

性能：与原生代码差距约 1.2~1.8×
```

### 4.3 wazero

用纯 Go 编写的 WASM 运行时，零依赖。

```
特点：
  - 不依赖任何 C 库，纯 Go 实现
  - 适合嵌入 Go 应用中
  - 部署简单（就是一个 Go 库）

性能：比 Wasmtime 慢，约 2~3×（无 JIT）
```

### 4.4 V8（浏览器）

Chrome、Node.js 使用的 JavaScript 引擎，也支持 WASM。

```
特点：
  - 内置 Turbofan/Maglev 优化编译器
  - 在浏览器中 WASM 性能极佳（接近原生）
  - 不适合服务端场景（启动慢，内存大）
```

## 5. 性能差距：WASM vs 原生代码

Dandelion 论文（SOSP 2025）测量了 WASM 沙箱的实际性能开销：

```
基准测试（相对原生代码的开销）：

                    原生      Wasmtime    WasmEdge
─────────────────────────────────────────────────
矩阵乘法            1.0×        1.3×        1.4×
JSON 解析           1.0×        1.5×        1.6×
图像压缩(JPEG)      1.0×        1.4×        1.5×
密码学(SHA-256)     1.0×        1.2×        1.3×
系统调用密集型       1.0×        2.0×        2.5×

数据来源：Dandelion (SOSP 2025) 和相关基准
注：具体数值取决于编译优化、硬件等因素
```

开销来源分析：

```
1. 内存边界检查（每次内存访问）
   → 约增加 5~15% 开销
   → 硬件辅助检查（利用 mprotect）可将开销降至 ~2%

2. 间接调用的类型检查
   → 约增加 3~8% 开销

3. JIT 编译的不如 clang -O3 激进
   → WASM JIT 不知道用户代码的高层语义，优化空间有限

4. 与宿主机的接口调用（host function call）
   → 每次调用系统接口需要在 WASM 和宿主机之间切换上下文
   → 系统调用密集型程序开销更大
```

## 6. 语言级沙箱的局限

### 6.1 只能跑编译到 WASM 的代码

这是最根本的限制：

```
可以用 WASM 沙箱：
  C/C++（clang + wasm32-wasi target）
  Rust（rustup target add wasm32-wasi）
  Go（GOOS=wasip1 GOARCH=wasm）
  大多数静态语言

不能直接用 WASM 沙箱：
  Python（因为 CPython 是动态语言，本身是个 C 程序——
           可以把 CPython 编译到 WASM，但有很大局限）
  Ruby、PHP、Perl（同上）
  已有的编译好的原生二进制文件（不重新编译无法变成 WASM）
```

在 Shimmy 的场景中：用户提交任意代码，可能是任何语言，甚至是已编译好的二进制。WASM 无法处理这种情况。

### 6.2 运行时本身的漏洞

WASM 的安全性依赖于运行时（Wasmtime、V8）的正确实现。历史上存在多个 WASM 运行时漏洞：

```
CVE-2021-32629  Cranelift (Wasmtime JIT 后端) 代码生成漏洞
               → 特定 WASM 代码可触发越界写
               → 修复：更严格的代码生成验证

V8 (Chrome) 多个 WASM 漏洞
               → 用于 Pwn2Own 等比赛的浏览器逃逸
```

### 6.3 侧信道攻击

WASM 的内存隔离无法防止侧信道攻击：

```
Spectre 攻击（CVE-2017-5753）：
  WASM 程序可以利用 CPU 的分支预测和缓存行为
  读取同一进程中其他 WASM 模块的内存内容

  （因为所有 WASM 实例共享同一个进程的 CPU 缓存）
```

浏览器为了缓解 Spectre，降低了 JavaScript `performance.now()` 的精度，并减少了共享内存的使用。

### 6.4 WASI 接口的扩展带来新风险

WASI（WebAssembly System Interface）定义了 WASM 程序访问系统资源的标准接口。随着 WASI 支持的功能越来越多，新的安全风险也随之引入：

```
WASI Preview 1：
  仅文件系统访问（基于 capability）+ 时钟 + 随机数

WASI Preview 2（Component Model）：
  网络访问、异步 I/O、多线程...
  每增加一个接口，就增加了攻击面
```

## 小结

```
WASM = 可移植、安全的低级字节码

安全模型的三个支柱：
  1. 线性内存  → 程序只能访问自己的内存区域
  2. 无裸指针  → 所有内存访问带边界检查
  3. 类型系统  → 防止混淆函数指针和数据

主流运行时：
  Wasmtime  → 生产就绪，Rust 编写，强安全性
  WasmEdge  → 边缘计算，启动快
  wazero    → 纯 Go，零依赖

性能：约 1.2~2× 原生代码
（远优于 DBI 的 2~3×，更远优于 ptrace 的 10×）

局限：
  只能跑编译到 WASM 的代码
  运行时本身可能有漏洞
  无法防止侧信道攻击
  通用性差（无法处理已有二进制）
```

## 与 Shimmy/沙箱设计的联系

WASM 是 Shimmy Sandbox 在设计阶段考虑过的方案之一（以及 Faasm 论文中探索的路线）。Faasm（ATC 2020）将用户函数编译为 WASM，获得了非常低的沙箱开销（接近原生性能）和快速的实例化速度。

然而，Shimmy 的目标是支持**任意语言和已有二进制**，不要求用户将代码重新编译为 WASM。这使得 WASM 方案不适用于 Shimmy 的核心需求，但 WASM 仍然是未来某些特定场景（如可信预编译函数库）的潜在优化方向。

Dandelion 论文（SOSP 2025）是研究 WASM 在云函数场景下性能的重要参考，书中引用了其性能数据。
