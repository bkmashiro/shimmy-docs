第七章：二进制插桩 #

本章介绍二进制插桩（Binary Instrumentation）——一种在不修改源代码的情况下，监控和修改程序行为的技术。这是 DynamoRIO 的理论基础，也是 Shimmy Sandbox 绕过 Lambda 限制的核心手段。

7.1 什么是"插桩" #

插桩（instrumentation）是一个来自测量领域的词：在被测量的对象上添加传感器，在不影响其正常工作的情况下获取其内部状态。

在软件领域，插桩意味着：在程序中插入额外的代码，用于监控、测量或修改程序的行为。

常见的插桩例子：

  • 添加 printf 调试程序（手动插桩）
  • 代码覆盖率工具（gcov、llvm-cov）
  • 性能分析器（profiler，如 perf、gprof）
  • 内存错误检测器（Valgrind、AddressSanitizer）
  • 沙箱系统（DynamoRIO、Pin）

7.2 静态插桩 vs 动态插桩 #

静态插桩（Static Instrumentation） #

静态插桩在程序运行之前修改程序——要么修改源代码，要么修改编译后的二进制文件。

源代码    静态插桩    插桩后源代码    编译器    插桩后可执行文件
──────► (源码插桩) ────────────► ──────► ───────────────────
   或
二进制    静态插桩    插桩后二进制
──────► (二进制重写) ──────────────────────────────────────

编译器内置的 gcov 代码覆盖率工具就是静态插桩：编译时 GCC 在每个基本块（后面会解释）插入计数器更新代码。

局限：

  • 需要源代码（或可以修改的二进制）
  • 无法处理运行时动态生成的代码（JIT）
  • 需要提前知道要插桩什么

动态插桩（Dynamic Binary Instrumentation，DBI） #

动态插桩在程序运行时修改程序——接管程序的执行，实时翻译机器码并插入分析代码。

原始程序 ──────────────────────────────────────────────►
              ↑ DBI框架接管！原始代码从不直接执行

         ┌─────────────────────────────────────────────┐
         │              DBI 框架的工作                  │
         │                                              │
         │  读取原始代码 → 分析 → 插入额外代码 → 执行   │
         └─────────────────────────────────────────────┘
              ↓
         修改后的代码在 "代码缓存" 中执行

优点：

  • 无需源代码
  • 覆盖所有代码（包括动态生成的代码、共享库）
  • 可以在运行时做出基于程序状态的决策

主要 DBI 框架：

  • DynamoRIO：由 MIT 开发，现由 Google 维护（Shimmy 使用）
  • Pin：由 Intel 开发，不开源
  • Valgrind：主要用于内存错误检测
  • QEMU TCG：用于处理器模拟（Proto E）

7.3 机器码的本质 #

在理解 DBI 之前，需要理解它操作的对象：机器码。

x86-64 变长指令编码 #

x86-64 指令不是固定长度的——长度从 1 字节到 15 字节不等。这使得解码（从字节流识别指令边界）非常复杂：

字节序列（十六进制）   对应的汇编指令
────────────────────  ──────────────────────────────────────
90                    nop                        (1 字节)
89 C8                 mov eax, ecx              (2 字节)
48 89 C8              mov rax, rcx              (3 字节，REX 前缀)
48 8B 45 F8           mov rax, [rbp-8]          (4 字节，内存寻址)
48 B8 EF CD AB 89    movabs rax, 0x123456789ABCDEF  (10 字节，64位立即数)
67 48 8D 04 25 00 00 00 00  lea rax, [0]       (9 字节，带前缀)
0F 05                 syscall                   (2 字节！关键！)

syscall 指令是 2 字节：0F 05。DynamoRIO 在翻译代码时，会识别所有出现的 0F 05 字节对，并在它们前面插入钩子代码。

用 objdump 查看真实的机器码 #

bash
# 反汇编 /bin/ls 的代码段
$ objdump -d /bin/ls | head -50
/bin/ls:     file format elf64-x86-64

Disassembly of section .text:

0000000000005540 <main@@Base>:
    5540:       f3 0f 1e fa             endbr64
    5544:       41 57                   push   r15
    5546:       41 56                   push   r14
    5548:       41 55                   push   r13
    554a:       41 54                   push   r12
    554c:       55                      push   rbp
    554d:       53                      push   rbx
    554e:       48 83 ec 78             sub    rsp,0x78
    5552:       48 8d 3d 67 bc 01 00    lea    rdi,[rip+0x1bc67]
    5559:       e8 e2 f5 ff ff          call   4b40 <setlocale@plt>
    ...

每行格式：地址: 字节序列 汇编指令

7.4 基本块（Basic Block） #

基本块（Basic Block）是动态二进制插桩框架操作的基本单元。

定义：一个基本块是一段满足以下条件的指令序列：

  • 只有一个入口（从第一条指令进入）
  • 只有一个出口（以分支/跳转/调用/返回/syscall 结束）
  • 中间没有任何分支指令
例子：以下代码片段                  对应的基本块划分

  test:                           ┌─ 基本块 1 ─────────┐
    cmp  rax, 0                   │   cmp  rax, 0       │
    jz   .zero                   │   jz   .zero        │← 出口（条件跳转）
                                  └────────────────────┘
  .nonzero:                       ┌─ 基本块 2 ─────────┐
    add  rax, 1                   │   add  rax, 1       │
    jmp  .end                     │   jmp  .end         │← 出口（无条件跳转）
                                  └────────────────────┘
  .zero:                          ┌─ 基本块 3 ─────────┐
    sub  rax, 1                   │   sub  rax, 1       │
    nop                           │   nop               │
                                  └────────────────────┘
  .end:                           ┌─ 基本块 4 ─────────┐
    ret                           │   ret               │← 出口（返回）
                                  └────────────────────┘

为什么以基本块为单位？

因为基本块内部的指令是线性执行的，没有跳转，可以安全地整体翻译。翻译一个基本块后，把翻译结果存入代码缓存（code cache），下次执行同一个基本块时直接复用缓存，避免重复翻译。

7.5 控制流图（Control Flow Graph） #

控制流图（Control Flow Graph，CFG）把所有基本块作为节点，把跳转关系作为边，形成有向图。

函数 factorial(n) 的 CFG:

  ┌─────────────────────────────┐
  │ 入口块: n = 参数             │
  │ cmp n, 0                    │
  │ jle .base_case              │──── 是 ────► ┌─────────────────┐
  └─────────────────────────────┘              │ .base_case:     │
         │ 否                                  │ mov eax, 1      │
         ▼                                     │ ret             │
  ┌─────────────────────────────┐              └─────────────────┘
  │ 递归块: sub n, 1             │
  │ call factorial              │
  │ imul eax, [原来的 n]         │
  │ ret                         │
  └─────────────────────────────┘

DBI 框架通过实时构建 CFG 来理解程序的控制流，从而正确插入分析代码。

7.6 动态二进制翻译器的工作流程 #

以 DynamoRIO 为例，整体工作流程如下：

程序启动时:
1. DynamoRIO 劫持程序的入口点
2. DynamoRIO 自己首先运行（初始化代码缓存、注册事件钩子）
3. 程序的原始代码永远不会直接执行！

执行循环:
┌──────────────────────────────────────────────────────────────┐
│                      DynamoRIO 调度器                         │
│                                                              │
│  1. 下一个基本块的地址是什么？（从 RIP 或上一个块的跳转目标）  │
│                                                              │
│  2. 代码缓存命中？                                           │
│     是 → 直接执行缓存中的翻译版本                            │
│     否 → 进入翻译阶段                                        │
│                                                              │
│  3. 翻译阶段:                                                │
│     a. 从原始地址解码指令                                    │
│     b. 对每条指令调用注册的"指令插桩"回调                    │
│     c. 如果遇到 syscall 指令: 替换为调用钩子函数              │
│     d. 修复跳转地址（原始地址→代码缓存地址）                 │
│     e. 写入代码缓存                                          │
│                                                              │
│  4. 执行翻译后的代码块                                       │
│     执行结束（遇到跳转/调用/返回/syscall）                   │
│     → 回到步骤 1                                             │
└──────────────────────────────────────────────────────────────┘

关键点：程序的原始机器码被永久存在原始地址，但永远不执行。CPU 始终在代码缓存中执行翻译后的代码。

7.7 代码缓存（Code Cache） #

代码缓存是 DynamoRIO 存放翻译后代码的内存区域，位于 DynamoRIO 自己管理的内存段中，不暴露给被插桩的程序。

内存布局（DynamoRIO 运行时）:

原始程序内存（只读，用于读取指令）:
0x400000 - 0x410000: .text（程序原始代码，DynamoRIO 从这里读，但 CPU 不从这里执行）

DynamoRIO 自己的内存（程序不可见）:
0x70000000 - 0x70100000: DynamoRIO 代码缓存
  ├── [基本块 0x400100 的翻译版本]
  │     mov rax, 1
  │     call DR_pre_syscall_hook    ← 插入的钩子调用
  │     syscall                     ← 原始 syscall（钩子决定是否执行）
  │     jmp [DR_dispatcher]        ← 替换原来的跳转
  ├── [基本块 0x400200 的翻译版本]
  └── ...

0x70100000 - 0x70200000: DynamoRIO 堆（自己的数据结构）
0x70200000 - 0x70300000: DynamoRIO 代码（DynamoRIO 自身的 .text）

7.8 覆盖范围：为什么 DBI 能覆盖所有代码 #

这是 DBI 相比其他方案最大的优势：

代码来源LD_PRELOADptraceDynamoRIO
动态链接库（libc 等）✓（可覆盖符号）✓✓
静态链接库（内嵌在程序中）✗ 无法✓✓
内联汇编（直接 syscall 指令）✗ 无法✓✓
JIT 生成的代码（V8、JVM）✗ 无法✓✓（mmap 后翻译）
自修改代码（较罕见）✗✓✓（失效后重新翻译）

DBI 在机器码指令级别操作，不关心代码是如何生成的。只要 CPU 要执行某段代码，DBI 就能看到并处理它。

7.9 DBI 的实现挑战 #

实现一个正确的 DBI 框架非常困难，以下是主要挑战：

挑战 1：自修改代码（Self-Modifying Code） #

有些程序在运行时修改自己的代码（比如旧版 JIT 编译器、反调试技术）。DBI 框架把基本块缓存后，如果原始代码被修改了，缓存就失效了。

DynamoRIO 的解决方案：监控程序的内存写入（通过 mprotect 钩子），当检测到代码页被修改时，使对应的缓存条目失效，下次执行时重新翻译。

挑战 2：JIT 编译器（Just-In-Time Compiler） #

V8（JavaScript）、HotSpot（Java）这类 JIT 编译器在运行时：

  1. 分配内存：mmap(PROT_READ|PROT_WRITE)
  2. 写入机器码
  3. 设置为可执行：mprotect(PROT_EXEC) 或直接 mmap(PROT_READ|PROT_WRITE|PROT_EXEC)
  4. 跳转执行

DynamoRIO 监控 mmap(PROT_EXEC) 和 mprotect(PROT_EXEC) 调用，在程序跳转到 JIT 代码时重新翻译。

但这产生了一个安全问题：如果 JIT 代码在生成时包含了 syscall 指令，而 DynamoRIO 还没翻译那个代码页，那么 JIT 代码就可能直接执行未被拦截的 syscall。

Shimmy 的解决方案：完全阻断 mmap(PROT_EXEC)。这会禁用 JIT，导致 JavaScript 等语言性能大幅下降，但保证了安全性。对于需要运行 JS 的场景，Shimmy 改用 Goja 引擎（第九章）。

挑战 3：多线程 #

多线程程序中，每个线程都需要自己的翻译上下文（drcontext）。代码缓存是共享的（节省内存），但必须用锁保护。同时，DBI 框架必须正确处理线程创建（clone() 系统调用）并为新线程初始化翻译环境。

挑战 4：信号处理器（Signal Handlers） #

信号（如 SIGSEGV、SIGTERM）会异步打断程序执行，跳转到信号处理函数。信号处理函数本身也是机器码，DBI 框架必须翻译它。

DynamoRIO 通过拦截 sigaction() 系统调用，将信号处理器包装在自己的翻译框架内。

7.10 DBI 与其他方案的完整对比 #

方案对比表：

                 LD_PRELOAD    ptrace      DynamoRIO
─────────────────────────────────────────────────────────
工作级别         库函数级      系统调用级  机器码指令级
覆盖范围         仅动态链接    全覆盖      全覆盖
静态链接程序     ✗            ✓           ✓
内联 syscall     ✗            ✓           ✓
JIT 生成代码     ✗            ✓           ✓（需要特殊处理）
性能开销         ~0%          ~900%       ~20-30%（生产CPU）
                                         ~67%（GitHub Actions）
实现复杂度       低            中          极高
需要特权         否            否（父子）  否
Lambda 可用      ✓            ✗（EPERM）  ✓
内存开销         ~0           低           ~50MB（代码缓存）
读取参数内存     ✓（指针）    ✓（完整）   ✓（dr_safe_read）
阻断系统调用     ✓            ✓           ✓
修改参数         ✓（拦截函数）✓（SETREGS）✓（dr_syscall_set_param）

结论一目了然：DynamoRIO 是在 Lambda 约束下，覆盖最全面、性能最可接受的方案。

7.11 性能：为什么 DBI 只慢 20-30% #

一个合理的疑问：如果每条指令都要被"翻译"，为什么不是慢 10 倍？

原因 1：代码缓存命中率极高

程序的热点代码（循环、常用函数）会被反复执行。一旦翻译并缓存，后续执行直接运行缓存版本，速度接近原生。翻译只发生在第一次执行某个基本块时。

原因 2：现代处理器的分支预测

翻译后的代码经过优化，分支目标已经是缓存中的地址，处理器的分支预测器可以正常工作。

原因 3：大部分指令不需要修改

大多数指令（算术、逻辑运算、数据移动）可以直接复制到代码缓存，完全不需要修改。只有遇到 syscall、间接跳转（jmp rax）、函数调用等才需要特殊处理。

DynamoRIO 的实测开销：

  • 空循环（无 syscall）：约 5-10% 开销
  • 普通计算密集型代码：约 20-30% 开销
  • I/O 密集型代码（频繁 syscall）：约 50-70% 开销
  • Shimmy 在 GitHub Actions 上测量：+67%（调度器噪音使结果偏高）
  • 在 Lambda vCPU 上预计：+20-30%

对于代码评测场景（学生代码通常有足够的时间限制裕量），20-30% 的开销是完全可以接受的。

小结 #

二进制插桩的核心概念:

插桩 = 在程序中插入监控/控制代码

静态插桩: 运行前修改
动态插桩 (DBI): 运行时翻译
    ↓
基本块: DBI 的操作单元（线性指令序列，单一出口）
代码缓存: 翻译结果的存储区
调度器: 维护"当前执行地址→缓存地址"的映射

DBI 的覆盖范围优势:
    • 静态链接 ✓
    • 内联汇编 ✓  
    • JIT 代码 ✓（mmap/mprotect 监控）

DBI vs ptrace: 相似的覆盖范围，但 DBI 快 10-15 倍

在 Lambda 中:
    ptrace → EPERM
    DynamoRIO → ✓（只需 mprotect(PROT_EXEC)，用户态完成）
