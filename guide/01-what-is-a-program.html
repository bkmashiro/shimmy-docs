第一章：什么是程序 #

本章从最基础的概念出发：一个程序究竟是什么？CPU 如何执行代码？理解这些，是理解沙箱技术的起点。

1.1 从人类语言到机器语言 #

你写下一行 Python 代码：

python
print("Hello, World!")

这行文字对计算机来说毫无意义。计算机内部只有一种东西：电——要么有电（1），要么没电（0）。

什么是比特（Bit） #

比特（bit，binary digit 的缩写）是计算机信息的最小单位。一个比特只有两种状态：0 或 1。

你可以把比特想象成一个灯开关：关着是 0，开着是 1。

什么是字节（Byte） #

字节（byte）是 8 个比特组合在一起。8 个比特可以表示 2⁸ = 256 种不同的状态（从 00000000 到 11111111）。

一个字节可以表示：

  • 一个 ASCII 字符（比如字母 'A' = 01000001 = 十进制 65）
  • 一个 0~255 之间的整数
  • 其他任何你约定好的含义
比特:  0  1  0  0  0  0  0  1
位置:  7  6  5  4  3  2  1  0
值:    0 +64+ 0 + 0 + 0 + 0 + 0 + 1 = 65 = 'A'

什么是指令（Instruction） #

指令是 CPU 能直接执行的最小操作单元。一条指令告诉 CPU 做一件具体的事，比如：

  • 把某个数字放进寄存器
  • 把两个寄存器的值相加
  • 跳转到另一个地址继续执行

指令在内存中也是以字节序列的形式存储的。x86-64 架构的指令长度从 1 字节到 15 字节不等。

1.2 什么是 RAM（内存） #

RAM（Random Access Memory，随机存取内存）是计算机在运行时临时存储数据的地方。

可以把 RAM 想象成一排有编号的格子，每个格子存放 1 个字节：

地址:   0x0000   0x0001   0x0002   0x0003   0x0004  ...
内容:  [  48  ] [  65  ] [  6C  ] [  6C  ] [  6F  ] ...
字符:     'H'      'e'      'l'      'l'      'o'

每个格子有一个唯一的地址（address）——这个地址就是它的编号，也是一个数字。在 64 位系统上，地址是 64 位（8 字节）的整数，理论上可以寻址 2⁶⁴ 字节（约 16 EB）的内存空间。

RAM 的特性：

  • 随机访问：可以直接通过地址访问任意位置，速度相同（相比硬盘的顺序访问）
  • 易失性：断电后数据消失
  • 速度：比硬盘快几千倍，但比 CPU 寄存器慢几十倍

1.3 CPU 是什么，它如何执行指令 #

CPU（Central Processing Unit，中央处理器）是计算机的"大脑"，负责执行指令。

寄存器（Register） #

寄存器是 CPU 内部的超小容量、超高速存储单元。寄存器不在 RAM 中——它们直接集成在 CPU 芯片上，访问速度比 RAM 快约 100 倍。

在 x86-64 架构（64 位 Intel/AMD 处理器）中，有 16 个通用寄存器，每个 64 位（8 字节）：

寄存器全名常见用途
RAXAccumulator函数返回值；系统调用号
RBXBase通用存储，被调用者保存
RCXCounter循环计数；第 4 个函数参数
RDXData第 3 个函数参数；I/O 操作
RSISource Index第 2 个函数参数；字符串源地址
RDIDestination Index第 1 个函数参数；字符串目标地址
RSPStack Pointer栈指针：始终指向当前栈顶
RBPBase Pointer栈帧基址指针
RIPInstruction Pointer程序计数器：下一条要执行的指令地址
R8–R15通用寄存器第 5–6 个函数参数；通用存储

其中最重要的两个：

  • RIP（Instruction Pointer）：始终存放下一条要执行的指令的内存地址。CPU 每执行完一条指令，就自动更新 RIP。
  • RSP（Stack Pointer）：始终指向栈（stack）的顶部。栈是函数调用时用来保存局部变量和返回地址的特殊内存区域。

取指-译码-执行循环（Fetch-Decode-Execute Cycle） #

CPU 的工作方式极其简单，就是不停地重复以下三步：

┌─────────────────────────────────────────────────────────────────┐
│                     CPU 主循环                                   │
│                                                                  │
│   1. FETCH（取指）                                               │
│      从 RIP 指向的内存地址读取指令字节                           │
│      RIP += 指令长度                                             │
│                                                                  │
│   2. DECODE（译码）                                              │
│      解析这些字节，确定是什么操作（加法？跳转？）               │
│      确定操作数（用哪些寄存器？用什么立即数？）                 │
│                                                                  │
│   3. EXECUTE（执行）                                             │
│      实际执行操作                                               │
│      如果是跳转指令，修改 RIP                                    │
│      回到步骤 1                                                  │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

举个例子，CPU 执行加法：

内存地址 0x1000: 48 01 F8    ; add rax, rdi  (把 RDI 的值加到 RAX)

1. FETCH: 读取字节 48 01 F8，RIP 变为 0x1003
2. DECODE: 这是 "add reg64, reg64" 操作，目标=RAX，源=RDI
3. EXECUTE: RAX = RAX + RDI

└──────────────────────────────────────────────────────────────────┘

1.4 从源代码到可执行文件 #

现在理解了 CPU 只能执行字节序列，那么 Python 代码是怎么变成 CPU 指令的？

编译型语言（Compiled Languages） #

编译器（compiler）是一个程序，它把人类写的源代码翻译成机器码（CPU 指令的字节序列）。

源代码 (C语言)                编译器                机器码
┌──────────────┐    ┌──────────────────────┐    ┌──────────────┐
│ int add(     │    │  词法分析              │    │ 55           │
│   int a,     │───►│  语法分析              │───►│ 48 89 E5     │
│   int b) {   │    │  语义分析              │    │ 89 7D FC     │
│   return a+b;│    │  优化                  │    │ 89 75 F8     │
│ }            │    │  代码生成              │    │ 8B 45 FC     │
└──────────────┘    └──────────────────────┘    │ 03 45 F8     │
                                                 │ 5D           │
                                                 │ C3           │
                                                 └──────────────┘

编译后的机器码可以直接被 CPU 执行，不需要任何中间软件。C、C++、Rust、Go 都是编译型语言。

解释型语言（Interpreted Languages） #

解释器（interpreter）是一个程序，它在运行时逐行读取源代码并执行，不生成独立的机器码文件。

源代码 (Python)          Python 解释器 (CPython)        CPU 执行
┌──────────────┐    ┌─────────────────────────────┐
│ print("Hi")  │───►│ 1. 读取 "print"              │
└──────────────┘    │ 2. 查找 print 内置函数        │───► CPU 执行
                    │ 3. 读取 "Hi" 字符串           │    解释器自己
                    │ 4. 调用 print 的 C 实现       │    的机器码
                    └─────────────────────────────┘

Python、JavaScript（无 JIT）、Ruby 是典型的解释型语言。

JIT 编译（Just-In-Time Compilation） #

现代 JavaScript 引擎（如 V8）和 Java 虚拟机（JVM）使用 JIT 编译：先解释执行，对热点代码即时编译成机器码。这在本文档第七章（二进制插桩）中会产生重要影响。

1.5 ELF 可执行文件格式 #

当 C 编译器编译完代码，生成的不是裸机器码，而是一个 ELF 文件（Executable and Linkable Format，可执行与可链接格式）。ELF 是 Linux 系统上的标准可执行文件格式。

ELF 魔数（Magic Bytes） #

ELF 文件的前 4 个字节始终是：7F 45 4C 46，即 \x7fELF。操作系统通过这 4 个字节识别这是一个 ELF 文件。

bash
$ xxd /bin/ls | head -2
00000000: 7f45 4c46 0201 0100 0000 0000 0000 0000  .ELF............

ELF 的主要节（Sections） #

ELF 文件被分成多个节（section），每个节存放不同类型的数据：

ELF 文件结构
┌─────────────────────────────┐
│ ELF Header (魔数、架构信息) │  ← 64 字节
├─────────────────────────────┤
│ .text  (代码节)              │  ← 编译出的机器码指令
│                             │     CPU 直接执行这里的字节
├─────────────────────────────┤
│ .rodata (只读数据节)         │  ← 字符串常量如 "Hello\n"
│                             │     运行时只读
├─────────────────────────────┤
│ .data  (已初始化数据节)      │  ← 全局变量（有初始值的）
│                             │     如 int x = 5;
├─────────────────────────────┤
│ .bss   (未初始化数据节)      │  ← 全局变量（初值为 0 的）
│                             │     文件中不占空间，加载时清零
├─────────────────────────────┤
│ .symtab (符号表)             │  ← 函数名、变量名到地址的映射
├─────────────────────────────┤
│ .dynamic (动态链接信息)      │  ← 依赖哪些共享库
└─────────────────────────────┘
  • .text：程序的机器码。当程序运行时，CPU 的 RIP 会在这里跳来跳去。
  • .rodata（Read-Only Data）：只读数据，比如字符串字面量 "Hello"。
  • .data：已初始化的全局变量和静态变量。
  • .bss（Block Started by Symbol）：未初始化的全局变量。ELF 文件里不真正存这些零，加载时由操作系统负责清零。

用 readelf 查看 ELF #

bash
$ readelf -h /bin/ls
ELF Header:
  Magic:   7f 45 4c 46 02 01 01 00 ...
  Class:                             ELF64
  Data:                              2's complement, little endian
  Machine:                           Advanced Micro Devices X86-64
  Entry point address:               0x6b10
  ...

$ readelf -S /bin/ls | head -20
Section Headers:
  [Nr] Name              Type             Address
  [ 0]                   NULL
  [ 1] .interp           PROGBITS         0000000000000318
  [ 2] .note.gnu...      NOTE             0000000000000338
  [14] .text             PROGBITS         0000000000005540
  [17] .rodata           PROGBITS         0000000000020260
  [25] .data             PROGBITS         0000000000036090
  [26] .bss              NOBITS           00000000000362a0

1.6 静态链接 vs 动态链接 #

程序通常需要使用别人写好的函数（比如 printf、malloc）。这些函数通常打包在库（library）中。有两种方式把库代码纳入程序：

静态链接（Static Linking） #

静态链接在编译时把库的代码直接复制进可执行文件。

┌────────────────────────────────┐
│       最终可执行文件           │
│  ┌──────────────────────────┐  │
│  │ 你的代码 (main.o)        │  │
│  ├──────────────────────────┤  │
│  │ libc 中 printf 的实现    │  │ ← 直接复制进来
│  ├──────────────────────────┤  │
│  │ libc 中 malloc 的实现    │  │ ← 直接复制进来
│  └──────────────────────────┘  │
└────────────────────────────────┘
文件较大，但运行时不依赖任何外部库

静态链接的可执行文件在运行时完全独立，不依赖系统上安装的任何库。Go 和 Rust 程序默认静态链接（或接近静态）。

动态链接（Dynamic Linking） #

动态链接在运行时才加载库（.so 文件，Shared Object）。可执行文件中只有"这里需要调用 printf"这样的引用，不包含实际代码。

┌─────────────────┐      运行时加载      ┌──────────────────────┐
│  你的程序 (小)  │ ─────────────────►  │  /lib/libc.so.6      │
│                 │                      │  (所有程序共享一份)  │
│  [需要printf]   │ ◄─────────────────  │  printf 在这里       │
└─────────────────┘      链接器解析      └──────────────────────┘

动态链接器（/lib64/ld-linux-x86-64.so.2）在程序启动时负责找到所需的 .so 文件并加载它们。

为什么这对沙箱技术至关重要 #

这里有一个关键的安全问题：

LD_PRELOAD（一种常见的沙箱技术）允许在程序加载时插入自定义库，覆盖标准库中的函数。例如，可以拦截 open() 函数的调用。

但这只对动态链接的程序有效！

  • 静态链接程序：库代码已经复制进可执行文件，没有动态加载过程，LD_PRELOAD 完全无效。
  • Go 程序：默认静态链接，所以 LD_PRELOAD 无法拦截 Go 程序的系统调用。
  • Rust 程序：类似 Go，通常静态链接关键组件。

这就是为什么 Shimmy Sandbox 不能依赖 LD_PRELOAD，而需要在更底层（机器码层面）进行拦截。

1.7 从源码到执行的完整流程 #

以一个简单的 C 程序为例，展示完整过程：

c
// hello.c
#include <stdio.h>

int main() {
    printf("Hello, World!\n");
    return 0;
}

源代码 hello.c
    │
    ▼ gcc -c hello.c
目标文件 hello.o (ELF, 含机器码但地址未确定)
    │
    ▼ gcc hello.o -o hello (链接器 ld)
可执行文件 hello (完整 ELF, 地址确定)
    │
    ▼ ./hello (shell 执行)
操作系统加载 ELF:
  1. 读取 ELF Header，找到程序入口点地址
  2. 创建新进程，分配虚拟地址空间
  3. 把 .text 映射到内存
  4. 把 .data/.rodata 映射到内存
  5. 如果是动态链接，启动动态链接器加载 .so
  6. 设置 RIP = 入口点地址
  7. 跳转！CPU 开始执行 _start (初始化) → main()
    │
    ▼ main() 执行
调用 printf("Hello, World!\n")
  → libc 的 printf 实现
    → 格式化字符串
    → 调用 write(1, "Hello, World!\n", 14) 系统调用
      → 进入内核
        → 内核把字节写入终端
      → 返回用户态
    → printf 返回
  → main 返回 0
→ _exit(0) 系统调用
  → 进程终止

注意最后两步出现了"系统调用"——这正是下一章和第三章要深入讲解的核心概念。

1.8 汇编语言：机器码的人类可读形式 #

汇编语言（Assembly Language）是机器码的人类可读表示，每条汇编指令对应一条（或极少数几条）机器码指令。

asm
; 这是 x86-64 汇编 (AT&T 语法)
; 计算 1+2 并把结果存入 RAX

mov $1, %rax      ; RAX = 1         (机器码: 48 C7 C0 01 00 00 00)
mov $2, %rdi      ; RDI = 2         (机器码: 48 C7 C7 02 00 00 00)
add %rdi, %rax    ; RAX = RAX + RDI (机器码: 48 01 F8)

汇编器（assembler）把汇编语言翻译成机器码。反汇编器（disassembler）做相反的事：把机器码翻译回汇编语言（比如 objdump -d）。

在 Shimmy Sandbox 的上下文中，DynamoRIO 本质上就是一个超级复杂的反汇编器+汇编器：它读取任意程序的机器码，理解其结构，在特定位置插入新指令，然后重新执行修改后的代码。

小结 #

本章建立了以下概念链：

比特 → 字节 → 指令 → 寄存器 → 内存
    ↓
源代码 → (编译器) → 机器码 → ELF 文件
    ↓
程序加载 → CPU 取指-译码-执行循环
    ↓
静态链接 vs 动态链接 → 影响 LD_PRELOAD 沙箱技术

核心要点：

  1. CPU 只执行字节序列（机器码），不懂 Python/C/Java
  2. RIP 寄存器指向下一条指令，CPU 盲目地跑完它
  3. 静态链接程序把所有代码打包进自身，绕过库层面的拦截
  4. ELF 格式把代码（.text）和数据（.data/.bss/.rodata）分开存放
